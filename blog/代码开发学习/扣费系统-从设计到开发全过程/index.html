一、项目背景

1.1 业务场景

这是一个算力租赁平台的扣费系统，主要业务场景包括：

  1. 包月预付费：用户购买包月算力，一次性扣除整月费用
  2. 按小时后付费：用户按小时使用算力，每小时自动续扣
  3. 退款处理：支持全额退款、部分退款、按订单退款、按扣费单退款
  4. 余额预警：提前通知用户余额不足，避免服务突然中断

1.2 核心挑战

  • 资金安全：不能重复扣费，不能多退少退
  • 高并发：支持大量用户同时扣费
  • 幂等性：网络重试、系统故障恢复时不能重复扣费
  • 系统解耦：与订单系统、资源系统解耦，通过消息队列异步通信
  • 可追溯性：每笔扣费都有完整的流水记录，支持对账

4.2 核心设计理念

4.2.1 主记录 + 明细记录设计

设计思路：

  • 主记录（DeductionRecord）：记录订单级别的扣费配置，每个订单只有1条主记录
  • 明细记录（DeductionDetail）：记录每次实际扣费的详细信息，每次扣费产生1条明细

优势：

  • 支持细粒度退款（可以退某一次扣费）
  • 支持部分退款（可以退部分金额）
  • 便于对账和审计

关系说明：

  • 一次性扣费：1个主记录 + 1个明细记录
  • 周期扣费：1个主记录 + N个明细记录（N=实际扣费次数）

4.2.2 同步调用 vs 消息队列

最终选择：同步调用

原因：

  1. 实时性要求高：扣费结果需要立即返回，不能异步处理
  2. 事务一致性：扣费、余额扣减、流水记录需要在同一事务中完成
  3. 简单可靠：同步调用逻辑清晰，易于调试和排查问题
  4. 消息队列用于通知：扣费成功后通过消息队列通知订单系统，而不是用于扣费流程本身

消息队列的使用场景：

  • 扣费结果通知（扣费成功/失败）
  • 余额不足通知
  • 余额预警通知
  • 退款结果通知

六、核心功能实现

6.1 扣费流程实现

6.1.1 核心代码

@Override
@Transactional(rollbackFor = Exception.class)
public DeductionResult deduct(DeductionRequest request) {
    log.info("[deduct][开始扣费] orderNo: {}, userId: {}, amount: {}, type: {}",
            request.getOrderNo(), request.getUserId(), request.getAmount(), request.getDeductionType());

    // 0. 参数校验
    deductionValidator.validateDeductionRequest(request);

    // 1. 生成幂等键
    String idempotentKey = generateIdempotentKey(request);

    // 2. 检查幂等性
    DeductionRecordDO existing = checkIdempotency(idempotentKey);
    if (existing != null) {
        // 如果是成功状态，直接返回
        if (DeductionStatusEnum.SUCCESS.getCode().equals(existing.getStatus())) {
            return DeductionResult.success(existing);
        }
        // 如果是处理中状态，返回处理中
        if (DeductionStatusEnum.PENDING.getCode().equals(existing.getStatus())) {
            return DeductionResult.fail("扣费处理中，请稍后查询结果");
        }
        // 如果是失败状态，允许重试
    }

    // 3. 创建扣费记录（处理并发冲突）
    DeductionRecordDO record;
    try {
        record = createDeductionRecord(request, idempotentKey);
    } catch (DuplicateKeyException e) {
        // 并发创建冲突，重新查询并返回
        existing = checkIdempotency(idempotentKey);
        if (existing != null && DeductionStatusEnum.SUCCESS.getCode().equals(existing.getStatus())) {
            return DeductionResult.success(existing);
        }
        return DeductionResult.fail("扣费处理冲突，请稍后重试");
    }

    // 4. 获取扣费前余额
    BigDecimal balanceBefore = getAccountBalance(request.getUserId());

    try {
        // 5. 扣减余额
        boolean success = accountBalanceService.reduceBalance(
                request.getUserId(),
                request.getAmount(),
                "订单扣费：" + request.getOrderNo()
        );

        if (!success) {
            throw exception(ACCOUNT_BALANCE_NOT_ENOUGH);
        }

        BigDecimal balanceAfter = getAccountBalance(request.getUserId());

        // 6. 更新扣费记录为成功
        updateDeductionSuccess(record, request);

        // 7. 创建扣费明细记录（首次扣费期数为1）
        createDeductionDetail(record, request.getAmount(), "SUCCESS", null, 1);

        // 8. 检查余额预警（仅周期扣费）
        if (Boolean.TRUE.equals(request.getIsPeriodic())) {
            balanceWarningService.checkBeforeDeduction(record, balanceAfter);
        }

        // 9. 发送扣费成功消息
        sendDeductionResultMessage(record, true, null);

        return DeductionResult.success(record);

    } catch (ServiceException e) {
        // 扣费失败处理
        updateDeductionFailed(record, e.getMessage());
        sendDeductionResultMessage(record, false, e.getMessage());
        
        if (ACCOUNT_BALANCE_NOT_ENOUGH.getCode().equals(e.getCode())) {
            sendBalanceInsufficientMessage(record, balanceBefore);
        }
        
        return DeductionResult.fail(e.getMessage());
    }
}

6.1.2 关键点说明

  1. 幂等性保证：

    • 通过 idempotent_key 唯一索引保证
    • 并发情况下捕获 DuplicateKeyException 后重新查询

  2. 事务管理：

    • 使用 @Transactional 保证扣费记录创建、余额扣减、明细记录创建在同一事务中

  3. 异常处理：

    • 业务异常（余额不足）和系统异常分别处理
    • 失败时更新记录状态并发送消息通知

6.2.2 续扣逻辑

@Override
public void executePeriodicDeductions() {
    // 1. 查询待续扣记录
    List<DeductionRecordDO> records = deductionRecordMapper.selectPendingPeriodicDeductions(
            LocalDateTime.now()
    );

    for (DeductionRecordDO record : records) {
        try {
            // 2. 计算漏扣的周期数（系统停机期间可能漏扣多个周期）
            int missedPeriods = calculateMissedPeriods(record);

            // 3. 逐个补扣漏掉的周期（最多补扣10个周期）
            int maxCatchUp = Math.min(missedPeriods, 10);
            for (int i = 0; i < maxCatchUp; i++) {
                boolean success = executePeriodicDeductionDirect(record);
                if (!success) {
                    // 扣费失败，停止继续补扣
                    break;
                }
            }
        } catch (Exception e) {
            log.error("[executePeriodicDeductions][续扣异常] orderNo: {}", record.getOrderNo(), e);
        }
    }
}

关键设计点：

  1. 停机补扣：

    • 计算漏扣的周期数
    • 最多补扣10个周期，避免一次性扣太多

  2. CAS更新：

    • 使用CAS更新防止与停止操作并发冲突
    • 更新失败时回滚余额

6.3 退款实现

6.3.1 按订单退款（粗粒度）

@Override
@Transactional(rollbackFor = Exception.class)
public RefundResult refundByOrderNo(String orderNo, String reason) {
    // 1. 查询可退款的扣费明细
    List<DeductionDetailDO> refundableDetails = deductionDetailMapper.selectRefundableByOrderNo(orderNo);
    
    BigDecimal totalRefundAmount = BigDecimal.ZERO;
    
    // 2. 统一执行退款
    for (DeductionDetailDO detail : refundableDetails) {
        BigDecimal alreadyRefunded = detail.getRefundedAmount() != null
                ? detail.getRefundedAmount()
                : BigDecimal.ZERO;
        BigDecimal refundableAmount = detail.getAmount().subtract(alreadyRefunded);
        
        if (refundableAmount.compareTo(BigDecimal.ZERO) <= 0) {
            continue;
        }
        
        // 执行退款
        BigDecimal refundAmount = executeRefundForDetail(detail, refundableAmount, reason);
        totalRefundAmount = totalRefundAmount.add(refundAmount);
        
        // 发送退款成功消息
        sendRefundResultMessageForDetail(detail, true, refundAmount, reason, false);
    }
    
    // 3. 停止主记录的周期扣费
    stopPeriodicDeductionByOrderNo(orderNo);
    
    return RefundResult.success(orderNo, totalRefundAmount);
}

6.3.2 部分退款（细粒度）

@Override
@Transactional(rollbackFor = Exception.class)
public RefundResult partialRefund(String deductionNo, BigDecimal refundAmount, String reason) {
    // 1. 查询主记录
    DeductionRecordDO record = deductionRecordMapper.selectByDeductionNo(deductionNo);
    
    // 2. 查询所有可退款的明细
    List<DeductionDetailDO> refundableDetails = deductionDetailMapper.selectRefundableByDeductionNo(deductionNo);
    
    // 3. 计算总的可退金额
    BigDecimal totalRefundableAmount = calculateTotalRefundableAmount(refundableDetails);
    
    // 4. 校验退款金额
    deductionValidator.validateRefundRequest(refundAmount, totalRefundableAmount);
    
    // 5. 按明细顺序执行部分退款（从最早的明细开始退）
    BigDecimal remainingRefund = refundAmount;
    BigDecimal actualRefundAmount = BigDecimal.ZERO;
    
    for (DeductionDetailDO detail : refundableDetails) {
        if (remainingRefund.compareTo(BigDecimal.ZERO) <= 0) {
            break;
        }
        
        BigDecimal detailRefundable = calculateDetailRefundable(detail);
        BigDecimal thisRefund = remainingRefund.min(detailRefundable);
        
        // 执行退款
        BigDecimal refunded = executeRefundForDetail(detail, thisRefund, reason);
        actualRefundAmount = actualRefundAmount.add(refunded);
        remainingRefund = remainingRefund.subtract(refunded);
    }
    
    return RefundResult.success(record.getOrderNo(), actualRefundAmount);
}

关键设计点：

  1. CAS更新防止并发退款：

    int updateCount = deductionDetailMapper.updateRefundedAmountAtomic(
        detail.getId(),
        totalRefundAmount,
        newRefundStatus,
        alreadyRefunded  // 期望值
);

if (updateCount == 0) {
    // CAS失败，回滚余额
    accountBalanceService.reduceBalance(...);
    throw exception(DEDUCTION_REFUND_CONCURRENT_CONFLICT);
}

  2. 部分退款逻辑：

    • 从最早的明细开始退款
    • 每个明细的可退金额 = 原金额 - 已退金额
    • 退款金额不能超过可退金额

七、关键技术点

7.1 幂等性保证

7.1.1 幂等键设计

private String generateIdempotentKey(DeductionRequest request) {
    if (Boolean.TRUE.equals(request.getIsPeriodic())) {
        // 周期扣费：幂等键不包含期数
        return String.format("%s_%s_PERIODIC",
            request.getOrderNo(),
            request.getDeductionType()
        );
    } else {
        // 一次性扣费
        return String.format("%s_%s_0",
                request.getOrderNo(),
                request.getDeductionType()
        );
    }
}

设计理由：

  • 周期扣费只创建一条主记录，后续续扣不创建新主记录，只创建明细记录
  • 因此幂等键不需要包含期数

7.1.2 并发处理

try {
    record = createDeductionRecord(request, idempotentKey);
} catch (DuplicateKeyException e) {
    // 并发情况下，另一个线程已创建，重新查询
    existing = checkIdempotency(idempotentKey);
    if (existing != null && DeductionStatusEnum.SUCCESS.getCode().equals(existing.getStatus())) {
        return DeductionResult.success(existing);
    }
    return DeductionResult.fail("扣费处理冲突，请稍后重试");
}

7.2.1 周期续扣的CAS更新

// 使用CAS更新主记录（防止与停止周期扣费并发冲突）

        new LambdaUpdateWrapper<>();
updateWrapper
        .set(DeductionRecordDO::getCurrentPeriod, nextPeriod)
        .set(DeductionRecordDO::getNextDeductionTime, nextDeductionTime)
        .set(DeductionRecordDO::getRetryCount, 0)
        .eq(DeductionRecordDO::getId, record.getId())
        .eq(DeductionRecordDO::getStatus, DeductionStatusEnum.SUCCESS.getCode()) // CAS条件
        .eq(DeductionRecordDO::getCurrentPeriod, record.getCurrentPeriod()); // CAS条件

    return false;
}

7.4 停机补扣机制

7.4.1 漏扣周期计算

private int calculateMissedPeriods(DeductionRecordDO record) {
    LocalDateTime nextDeductionTime = record.getNextDeductionTime();
    LocalDateTime now = LocalDateTime.now();

    if (nextDeductionTime == null || nextDeductionTime.isAfter(now)) {
        return 0;
    }

    // 计算时间差（小时）
    long hoursDiff = Duration.between(nextDeductionTime, now).toHours();

    // 计算漏扣周期数 = 时间差 / 周期小时数 + 1（当前周期）
    int periodHours = record.getPeriodHours() != null ? record.getPeriodHours() : 1;
    int missedPeriods = (int) (hoursDiff / periodHours) + 1;

    // 检查是否超过总周期限制
    if (record.getTotalPeriods() != null && record.getTotalPeriods() > 0) {
        int remainingPeriods = record.getTotalPeriods() - record.getCurrentPeriod();
        missedPeriods = Math.min(missedPeriods, remainingPeriods);
    }

    return missedPeriods;
}

7.4.2 补扣限制

// 最多补扣10个周期，避免系统长时间停机后一次性扣太多
int maxCatchUp = Math.min(missedPeriods, 10);

for (int i = 0; i < maxCatchUp; i++) {
    boolean success = executePeriodicDeductionDirect(record);
    if (!success) {
        // 扣费失败，停止继续补扣
        break;
    }
}

八、踩坑与优化

8.1 设计问题与改进

8.1.1 问题1：控制粒度太粗

问题描述：

  • 最初只支持按订单号退款和停止扣费
  • 升降配场景下，无法精确控制某个扣费单

解决方案：

  • 新增按扣费单号退款：refundByDeductionNo(deductionNo, reason)
  • 新增部分退款：partialRefund(deductionNo, refundAmount, reason)
  • 新增按扣费单号停止：stopPeriodicDeductionByDeductionNo(deductionNo)

8.1.2 问题2：周期扣费续扣时创建新主记录

问题描述：

  • 最初设计每次续扣都创建新的主记录
  • 导致一个订单有多条主记录，难以管理

解决方案：

  • 周期扣费只创建一条主记录
  • 后续续扣只创建明细记录，不创建新主记录
  • 主记录通过 current_period 和 next_deduction_time 跟踪续扣状态

8.1.3 问题3：退款并发冲突

问题描述：

  • 多个线程同时退款时，可能重复退款

解决方案：

  • 使用CAS更新退款金额
  • 更新失败时回滚余额并抛出异常

十、总结与思考

10.1 核心亮点

  1. 主记录+明细记录设计：

    • 支持细粒度退款和部分退款
    • 便于对账和审计

  2. 完善的幂等性保证：

    • 数据库唯一索引 + 应用层校验
    • 支持并发场景下的幂等性

  3. CAS更新防止并发冲突：

    • 周期续扣、退款等关键操作都使用CAS更新
    • 保证数据一致性

  4. 停机补扣机制：

    • 系统停机期间漏扣的周期自动补扣
    • 限制最多补扣10个周期，避免一次性扣太多

  5. 余额预警机制：

    • 提前3天和1天两级预警
    • Redis去重，避免重复发送

10.2 设计思考

10.2.1 为什么选择同步调用而不是消息队列？

原因：

  1. 实时性要求：扣费结果需要立即返回，不能异步处理
  2. 事务一致性：扣费、余额扣减、流水记录需要在同一事务中完成
  3. 简单可靠：同步调用逻辑清晰，易于调试和排查问题

消息队列的使用场景：

  • 扣费结果通知（扣费成功后通知订单系统）
  • 余额不足通知
  • 余额预警通知

10.2.2 为什么周期扣费只创建一条主记录？

原因：

  1. 便于管理：一个订单只有一条主记录，便于查询和管理
  2. 状态跟踪：通过 current_period 和 next_deduction_time 跟踪续扣状态
  3. 退款方便：退款时只需要操作一条主记录

明细记录的作用：

  • 记录每次实际扣费的详细信息
  • 支持细粒度退款（可以退某一次扣费）

10.3 未来优化方向

  1. 性能优化：

    • 批量处理续扣任务
    • Redis缓存热点数据
    • 数据库分表（按月分表）

  2. 功能增强：

    • 支持更多计费模式（按天、按周）
    • 支持优惠券抵扣
    • 支持欠费处理

  3. 监控完善：

    • 对接监控系统（Prometheus + Grafana）
    • 配置告警规则
    • 定期生成对账报表

10.4 经验总结

  1. 设计阶段要充分考虑业务场景：

    • 升降配、部分退款等复杂场景要在设计阶段考虑清楚

  2. 幂等性是金融系统的生命线：

    • 所有涉及资金的操作都必须保证幂等性
    • 数据库唯一索引 + 应用层校验是双重保障

  3. 并发控制要使用CAS而不是锁：

    • CAS更新性能更好，不会阻塞其他线程
    • 更新失败时要有回滚机制

  4. 异常处理要完善：

    • 业务异常和系统异常要分别处理
    • 失败时要记录详细日志，便于排查问题

  5. 测试要充分：

    • 单元测试、集成测试、压力测试都要覆盖
    • 特别是并发场景和异常场景的测试
