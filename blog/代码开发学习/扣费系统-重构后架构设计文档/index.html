一、重构背景

1.1 原有问题

在重构前，扣费系统的核心逻辑都集中在 DeductionServiceImpl 中，存在以下问题：

  1. 代码臃肿：单个类超过700行，方法过长，可读性差
  2. 职责不清：一个类承担了太多职责（扣费、退款、周期续扣、消息发送等）
  3. 难以测试：方法耦合度高，单元测试困难
  4. 难以维护：修改一个功能可能影响其他功能
  5. 异常处理分散：异常处理逻辑分散在各个方法中，难以统一管理

1.2 重构动机

  • 提升代码质量：通过模块化拆分，提高代码的可读性和可维护性
  • 降低耦合度：每个组件职责单一，便于独立测试和修改
  • 增强异常处理：统一异常记录机制，便于问题排查
  • 提高可扩展性：新功能可以独立开发，不影响现有功能

二、重构目标

2.1 设计原则

  1. 单一职责原则：每个组件只负责一个明确的功能
  2. 开闭原则：对扩展开放，对修改关闭
  3. 依赖倒置原则：依赖抽象而非具体实现
  4. 接口隔离原则：接口设计精简，避免冗余

2.2 重构目标

  • 将大方法拆分为多个小方法，每个方法不超过50行
  • 按功能职责拆分组件，每个组件职责单一
  • 统一异常处理机制，所有异常都记录到数据库
  • 提高代码可测试性，每个组件都可以独立测试
  • 保持原有功能不变，确保向后兼容

四、核心组件

4.1 DeductionServiceImpl（主服务）

职责：编排各个组件，实现扣费、退款、周期续扣等核心业务流程。

特点：

  • 作为门面，对外提供统一的服务接口
  • 负责事务管理（@Transactional）
  • 协调各个组件完成业务流程
  • 处理异常并记录到异常表

核心方法：

  • deduct() - 执行扣费
  • refundByOrderNo() - 按订单号退款
  • refundByDeductionNo() - 按扣费单号退款
  • partialRefund() - 部分退款
  • executePeriodicDeductions() - 执行周期续扣任务

4.2 DeductionRecordManager（记录管理器）

职责：管理扣费记录的创建、更新、幂等性检查等操作。

核心方法：

  • generateIdempotentKey() - 生成幂等键
  • checkIdempotency() - 检查幂等性
  • createDeductionRecord() - 创建扣费记录
  • updateDeductionSuccess() - 更新扣费记录为成功
  • createDeductionDetail() - 创建扣费明细记录

设计亮点：

  • 幂等键生成逻辑集中管理
  • 处理并发冲突（DuplicateKeyException）
  • 周期扣费相关字段的统一设置

4.3 RefundExecutor（退款执行器）

职责：执行单次退款的核心逻辑（增加余额、CAS更新明细记录）。

核心方法：

  • execute() - 执行退款

设计亮点：

  • 使用CAS更新防止并发退款冲突
  • 先增加余额，再更新明细（失败时事务回滚）
  • 支持部分退款（通过 refundedAmount 字段）

关键代码：

// 1. 先增加余额
accountBalanceService.addBalance(...);

// 2. CAS更新明细记录
int updateCount = deductionDetailMapper.updateRefundedAmountAtomic(
    detail.getId(),
    totalRefundAmount,
    newRefundStatus,
    alreadyRefunded  // 期望值
);

if (updateCount == 0) {
    // CAS失败，抛出异常触发事务回滚
    throw exception(DEDUCTION_REFUND_CONCURRENT_CONFLICT);
}

4.4 PeriodicDeductionProcessor（周期扣费处理器）

职责：处理周期扣费的核心逻辑，包括漏扣周期计算、周期续扣执行、重试机制等。

核心方法：

  • processOneRecord() - 处理单个订单的周期续扣
  • executeWithOneRetry() - 执行周期续扣（带1次重试）
  • calculateMissedPeriods() - 计算漏扣的周期数
  • handleSuccessPostActions() - 处理扣费成功后的非关键操作

设计亮点：

  • 支持停机补扣（最多补扣10个周期）
  • 每个周期的扣费在独立事务中执行
  • 失败时自动停止周期扣费
  • 非关键操作（预警、消息）失败不影响主流程

关键逻辑：

// 1. 计算漏扣周期数
int missedPeriods = calculateMissedPeriods(record);

// 2. 逐个补扣（最多10个周期）
int maxCatchUp = Math.min(missedPeriods, MAX_CATCH_UP_PERIODS);
for (int i = 0; i < maxCatchUp; i++) {
    PeriodicDeductionResult result = executeWithOneRetry(record, balanceBefore);
    if (!result.isSuccess()) {
        break; // 失败则停止
    }
}

4.5 PeriodicDeductionExecutor（周期扣费执行器）

职责：执行单个周期的扣费操作（扣减余额、更新记录、创建明细）。

特点：

  • 每个周期扣费在独立事务中执行（@Transactional）
  • 使用CAS更新防止并发冲突
  • 失败时回滚余额

4.6 PeriodicDeductionStopper（周期扣费停止器）

职责：停止周期扣费（更新状态为STOPPED）。

特点：

  • 使用CAS更新防止并发冲突
  • 支持按订单号停止和按扣费单号停止

4.7 DeductionMessageSender（消息发送器）

职责：发送各种业务消息（扣费结果、退款结果、余额不足、停止消息等）。

特点：

  • 消息发送失败不影响主流程
  • 失败时记录到异常表
  • 支持批量发送退款消息

4.8 DeductionExceptionService（异常记录服务）

职责：记录系统中所有需要人工干预的数据异常情况。

异常类型：

  • CAS_FAILED - CAS更新失败
  • TRANSACTION_ROLLBACK - 事务回滚
  • DATA_INCONSISTENCY - 数据不一致
  • OPERATION_FAILED - 操作失败

核心方法：

  • recordException() - 记录异常（通用方法）
  • recordCasFailed() - 记录CAS更新失败
  • recordBalanceReducedButFailed() - 记录余额已扣减但后续操作失败
  • recordStopFailed() - 记录停止周期扣费失败
  • recordMessageSendFailed() - 记录消息发送失败

设计亮点：

  • 统一异常记录机制，便于每日检查和处理
  • 记录数据快照，便于问题排查
  • 支持异常处理状态跟踪

4.9 其他组件

  • DeductionValidator：参数校验器，负责校验扣费请求和退款请求
  • BalanceWarningService：余额预警服务，负责余额预警检查和消息发送
  • AccountBalanceHelper：账户余额助手，提供余额查询的便捷方法（带重试）
  • RefundCalculator：退款计算工具类，计算可退金额
  • DeductionNoUtils：扣费单号生成工具类

五、关键实现

5.1 扣费流程（重构后）

@Override
@Transactional(rollbackFor = Exception.class)
public DeductionResult deduct(DeductionRequest request) {
    // 0. 参数校验
    deductionValidator.validateDeductionRequest(request);

    // 1. 生成幂等键
    String idempotentKey = deductionRecordManager.generateIdempotentKey(request);

    // 2. 检查幂等性
    DeductionRecordDO existing = deductionRecordManager.checkIdempotency(idempotentKey);
    if (existing != null && DeductionStatusEnum.SUCCESS.getCode().equals(existing.getStatus())) {
        return DeductionResult.success(existing);
    }

    // 3. 创建扣费记录
    DeductionRecordDO record = deductionRecordManager.createDeductionRecord(request, idempotentKey);

    // 4. 扣减余额
    BigDecimal balanceBefore = accountBalanceHelper.getAccountBalance(request.getUserId());
    accountBalanceService.reduceBalance(...);
    BigDecimal balanceAfter = accountBalanceHelper.getAccountBalance(request.getUserId());

    // 5. 更新扣费记录为成功
    deductionRecordManager.updateDeductionSuccess(record, request);

    // 6. 创建扣费明细记录
    deductionRecordManager.createDeductionDetail(record, request.getAmount(), "SUCCESS", null, 1);

    // 7. 余额预警检查（非关键操作）
    if (Boolean.TRUE.equals(request.getIsPeriodic())) {
        try {
            balanceWarningService.checkBeforeDeduction(record, balanceAfter);
        } catch (Exception e) {
            log.error("[deduct][余额预警检查失败]", e);
        }
    }

    // 8. 发送扣费成功消息（非关键操作）
    try {
        deductionMessageSender.sendDeductionResultMessage(record, true, null);
    } catch (Exception e) {
        log.error("[deduct][发送消息失败]", e);
        deductionExceptionService.recordMessageSendFailed(record, "扣费成功消息", e);
    }

    return DeductionResult.success(record);
}

重构亮点：

  • 每个步骤职责清晰，易于理解和维护
  • 非关键操作（预警、消息）失败不影响主流程
  • 异常统一记录到异常表

5.2 退款流程（重构后）

@Override
@Transactional(rollbackFor = Exception.class)
public RefundResult refundByOrderNo(String orderNo, String reason) {
    // 1. 查询可退款的扣费明细
    List<DeductionDetailDO> refundableDetails = deductionDetailMapper.selectRefundableByOrderNo(orderNo);

    BigDecimal totalRefundAmount = BigDecimal.ZERO;
    List<RefundMessageInfo> successfulRefunds = new ArrayList<>();

    // 2. 统一执行退款
    for (DeductionDetailDO detail : refundableDetails) {
        BigDecimal refundableAmount = RefundCalculator.calculateRefundableAmount(detail);
        if (refundableAmount.compareTo(BigDecimal.ZERO) <= 0) {
            continue;
        }

        // 执行退款（失败会抛异常触发回滚）
        BigDecimal refundAmount = refundExecutor.execute(detail, refundableAmount, reason);
        totalRefundAmount = totalRefundAmount.add(refundAmount);
        successfulRefunds.add(new RefundMessageInfo(detail, refundAmount, reason, false));
    }

    // 3. 停止周期扣费（关键操作）
    boolean stopped = periodicDeductionStopper.stopByOrderNo(orderNo);
    if (!stopped) {
        throw exception(DEDUCTION_UPDATE_STATUS_FAILED);
    }

    // 4. 发送退款消息（非关键操作）
    deductionMessageSender.sendRefundMessages(successfulRefunds, "refundByOrderNo");

    return RefundResult.success(orderNo, totalRefundAmount);
}

重构亮点：

  • 退款逻辑封装在 RefundExecutor 中，可复用
  • 退款计算逻辑封装在 RefundCalculator 中
  • 消息发送失败不影响退款结果

5.3 周期续扣流程（重构后）

@Override
public void executePeriodicDeductions() {
    // 1. 查询待续扣记录
    List<DeductionRecordDO> records = deductionRecordMapper.selectPendingPeriodicDeductions(LocalDateTime.now());

    for (DeductionRecordDO record : records) {
        try {
            // 2. 处理单个订单的周期续扣
            periodicDeductionProcessor.processOneRecord(record);
        } catch (Exception e) {
            // 3. 处理失败：停止周期扣费、发送消息、记录异常
            handlePeriodicDeductionFailure(record, e);
        }
    }
}

重构亮点：

  • 周期续扣逻辑封装在 PeriodicDeductionProcessor 中
  • 每个订单的处理完全独立，失败不影响其他订单
  • 异常统一处理，记录到异常表

六、异常处理机制

6.1 异常分类

系统将异常分为以下几类：

  1. 业务异常（ServiceException）

    • 余额不足、参数校验失败等
    • 需要回滚事务
    • 发送业务消息通知

  2. 系统异常（Exception）

    • 数据库连接失败、网络异常等
    • 需要回滚事务
    • 记录到异常表

  3. 非关键操作异常

    • 消息发送失败、余额预警失败等
    • 不影响主流程
    • 记录到异常表

6.2 异常记录机制

所有需要人工干预的异常都会记录到 tgkw_deduction_exception 表：

// 示例：记录CAS更新失败
deductionExceptionService.recordCasFailed(
    record,
    exceptionCode,
    "停止周期扣费CAS失败，可能导致重复扣费",
    exception
);

异常表字段：

  • business_module - 业务模块（DEDUCTION、REFUND等）
  • business_type - 业务类型（扣费、退款等）
  • exception_type - 异常类型（CAS_FAILED、TRANSACTION_ROLLBACK等）
  • exception_code - 异常错误码
  • exception_message - 异常信息
  • data_snapshot - 数据快照（JSON格式）
  • handle_status - 处理状态（UNHANDLED、HANDLED等）

6.3 异常处理流程

异常发生
    ↓
判断异常类型
    ↓
业务异常 → 发送业务消息 → 抛出异常 → 事务回滚
    ↓
系统异常 → 记录到异常表 → 抛出异常 → 事务回滚
    ↓
非关键操作异常 → 记录到异常表 → 继续执行

七、重构收益

7.1 代码质量提升

  • 可读性提升：每个组件职责单一，代码清晰易懂
  • 可维护性提升：修改一个功能不影响其他功能
  • 可测试性提升：每个组件都可以独立测试
  • 代码行数减少：主服务类从700+行减少到400+行

7.2 架构优化

  • 模块化设计：按功能职责拆分组件
  • 低耦合高内聚：组件之间依赖关系清晰
  • 易于扩展：新功能可以独立开发

7.3 异常处理增强

  • 统一异常记录：所有异常都记录到数据库
  • 便于问题排查：异常信息完整，包含数据快照
  • 支持异常处理跟踪：记录处理状态

7.4 性能优化

  • 非关键操作异步化：消息发送、预警检查失败不影响主流程
  • 事务粒度优化：周期续扣每个周期独立事务

八、总结与思考

8.1 重构经验总结

  1. 渐进式重构：不要一次性重构所有代码，按模块逐步重构
  2. 保持功能不变：重构过程中确保原有功能不受影响
  3. 充分测试：重构后要进行充分的单元测试和集成测试
  4. 文档同步更新：重构后要及时更新技术文档

8.2 设计原则应用

  • 单一职责原则：每个组件只负责一个明确的功能
  • 开闭原则：通过接口和抽象类实现扩展
  • 依赖倒置原则：依赖接口而非具体实现
  • 接口隔离原则：接口设计精简，避免冗余

8.3 未来优化方向

  1. 性能优化：

    • 批量处理周期续扣任务
    • Redis缓存热点数据
    • 数据库分表（按月分表）

  2. 功能增强：

    • 支持更多计费模式（按天、按周）
    • 支持优惠券抵扣
    • 支持欠费处理

  3. 监控完善：

    • 对接监控系统（Prometheus + Grafana）
    • 配置告警规则
    • 定期生成对账报表

  4. 异常处理优化：

    • 异常自动处理机制（如自动重试）
    • 异常处理工作流
    • 异常统计分析
