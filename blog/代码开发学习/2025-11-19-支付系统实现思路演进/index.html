支付系统实现思路演进

支付系统实现思路演进

实现一个支持多种支付渠道（微信、支付宝等）的支付系统

阶段一：最简单的实现（面向过程）

思路

直接写一个方法，根据渠道类型调用不同的支付逻辑

public class PaymentService {

    public String pay(String channelCode, PayRequest request) {
        if ("wx_native".equals(channelCode)) {
            // 微信Native支付
            WxPayClient wxClient = new WxPayClient();
            return wxClient.createOrder(request);
        } else if ("alipay_wap".equals(channelCode)) {
            // 支付宝WAP支付
            AlipayClient alipayClient = new AlipayClient();
            return alipayClient.createOrder(request);
        } else if ("wx_app".equals(channelCode)) {
            // 微信APP支付
            WxPayClient wxClient = new WxPayClient();
            return wxClient.createAppOrder(request);
        }
        // ... 更多 if-else
        throw new RuntimeException("不支持的支付渠道");
    }
}

问题

  1. 代码膨胀：每增加一个支付渠道，都要加一个 if-else
  2. 难以维护：所有支付逻辑混在一起
  3. 违反开闭原则：修改代码才能添加新功能

阶段二：面向对象初步改进

思路

每个支付渠道单独一个类，用接口统一

// 1. 定义接口
public interface PayClient {
    String createOrder(PayRequest request);
}

// 2. 实现类
public class WxNativePayClient implements PayClient {
    @Override
    public String createOrder(PayRequest request) {
        // 微信Native支付逻辑
        return "wx_native_order_result";
    }
}

public class AlipayWapPayClient implements PayClient {
    @Override
    public String createOrder(PayRequest request) {
        // 支付宝WAP支付逻辑
        return "alipay_wap_order_result";
    }
}

// 3. 使用
public class PaymentService {
    public String pay(String channelCode, PayRequest request) {
        PayClient client = null;
        if ("wx_native".equals(channelCode)) {
            client = new WxNativePayClient();
        } else if ("alipay_wap".equals(channelCode)) {
            client = new AlipayWapPayClient();
        }
        // ... 还是很多 if-else

        return client.createOrder(request);
    }
}

改进

  • 每个支付渠道独立成类，职责清晰
  • 通过接口统一调用方式

问题

  1. 还是有 if-else：创建客户端时还是需要判断
  2. 没有统一流程：每个实现类都要自己写验证、异常处理
  3. 重复代码：每个实现类都有类似的逻辑

阶段三：引入工厂模式（策略模式雏形）

思路

用 Map 存储策略映射，消除 if-else

public class PayClientFactory {

    // 策略映射表：渠道代码 → 客户端类
    private static final Map<String, Class<? extends PayClient>> CLIENT_MAP = new HashMap<>();

    static {
        CLIENT_MAP.put("wx_native", WxNativePayClient.class);
        CLIENT_MAP.put("alipay_wap", AlipayWapPayClient.class);
        CLIENT_MAP.put("wx_app", WxAppPayClient.class);
        // 新增渠道只需要在这里添加一行
    }

    public static PayClient createClient(String channelCode) {
        Class<? extends PayClient> clientClass = CLIENT_MAP.get(channelCode);
        if (clientClass == null) {
            throw new RuntimeException("不支持的支付渠道: " + channelCode);
        }

        try {
            return clientClass.newInstance();
        } catch (Exception e) {
            throw new RuntimeException("创建支付客户端失败", e);
        }
    }
}

// 使用
public class PaymentService {
    public String pay(String channelCode, PayRequest request) {
        PayClient client = PayClientFactory.createClient(channelCode);  // 没有 if-else 了！
        return client.createOrder(request);
    }
}

改进

  • 消除 if-else：通过 Map 查找，O(1) 时间复杂度
  • 易于扩展：新增渠道只需要在 Map 中添加一行
  • 符合开闭原则：对扩展开放，对修改关闭

问题

  1. 重复代码问题没解决：每个实现类都要写验证、异常处理
  2. 没有统一流程：不同实现类的流程可能不一致

阶段四：引入模板方法模式

思路

抽象类定义统一流程，子类实现具体逻辑

// 1. 抽象类定义模板方法
public abstract class AbstractPayClient implements PayClient {

    protected Long channelId;
    protected PayClientConfig config;

    public AbstractPayClient(Long channelId, PayClientConfig config) {
        this.channelId = channelId;
        this.config = config;
    }

    // 模板方法：定义统一的支付流程
    @Override
    public final String createOrder(PayRequest request) {
        // 步骤1：统一验证（所有渠道都验证）
        validateRequest(request);

        // 步骤2：执行支付（由子类实现）
        String result;
        try {
            result = doCreateOrder(request);
        } catch (Exception e) {
            // 步骤3：统一异常处理（所有渠道都这样处理）
            log.error("支付失败", e);
            throw new PayException("支付失败", e);
        }

        // 步骤4：统一后处理（所有渠道都这样）
        log.info("支付成功: {}", result);
        return result;
    }

    // 抽象方法：子类实现具体支付逻辑
    protected abstract String doCreateOrder(PayRequest request);

    // 统一验证逻辑
    private void validateRequest(PayRequest request) {
        if (request == null || request.getAmount() == null) {
            throw new IllegalArgumentException("请求参数错误");
        }
    }
}

// 2. 子类只需要实现核心逻辑
public class WxNativePayClient extends AbstractPayClient {

    public WxNativePayClient(Long channelId, WxPayClientConfig config) {
        super(channelId, config);
    }

    @Override
    protected String doCreateOrder(PayRequest request) {
        // 只需要实现微信支付的具体逻辑
        // 不需要写验证、异常处理、日志等
        WxPayClient wxClient = new WxPayClient(config);
        return wxClient.createNativeOrder(request);
    }
}

public class AlipayWapPayClient extends AbstractPayClient {

    public AlipayWapPayClient(Long channelId, AlipayPayClientConfig config) {
        super(channelId, config);
    }

    @Override
    protected String doCreateOrder(PayRequest request) {
        // 只需要实现支付宝支付的具体逻辑
        AlipayClient alipayClient = new AlipayClient(config);
        return alipayClient.createWapOrder(request);
    }
}

改进

  • 统一流程：所有支付渠道都遵循相同的执行步骤
  • 减少重复：验证、异常处理、日志等公共逻辑只在父类写一次
  • 职责清晰：子类只关注自己的支付逻辑

问题

  1. 工厂模式不完善：每次都创建新实例，没有缓存
  2. 配置管理：没有统一管理配置的更新

阶段五：完善工厂模式（加入缓存和配置管理）

思路

工厂管理客户端生命周期：创建、缓存、更新

public class PayClientFactory {

    // 策略映射表
    private static final Map<String, Class<? extends AbstractPayClient>> CLIENT_CLASS_MAP = new HashMap<>();

    // 客户端缓存（避免重复创建）
    private final Map<Long, AbstractPayClient> clientCache = new ConcurrentHashMap<>();

    static {
        CLIENT_CLASS_MAP.put("wx_native", WxNativePayClient.class);
        CLIENT_CLASS_MAP.put("alipay_wap", AlipayWapPayClient.class);
        // ...
    }

    /**
     * 获取或创建支付客户端
     */
    public PayClient getOrCreateClient(Long channelId, String channelCode, PayClientConfig config) {
        // 先从缓存获取
        AbstractPayClient client = clientCache.get(channelId);

        if (client == null) {
            // 缓存中没有，创建新实例
            client = createClient(channelId, channelCode, config);
            client.init();  // 初始化
            clientCache.put(channelId, client);  // 加入缓存
        } else {
            // 已存在，检查配置是否更新
            if (!config.equals(client.getConfig())) {
                client.refresh(config);  // 更新配置
            }
        }

        return client;
    }

    /**
     * 创建客户端实例（策略模式）
     */
    private AbstractPayClient createClient(Long channelId, String channelCode, PayClientConfig config) {
        // 1. 根据渠道代码获取对应的客户端类
        Class<? extends AbstractPayClient> clientClass = CLIENT_CLASS_MAP.get(channelCode);
        if (clientClass == null) {
            throw new RuntimeException("不支持的支付渠道: " + channelCode);
        }

        // 2. 反射创建实例
        try {
            Constructor<? extends AbstractPayClient> constructor =
                clientClass.getConstructor(Long.class, PayClientConfig.class);
            return constructor.newInstance(channelId, config);
        } catch (Exception e) {
            throw new RuntimeException("创建支付客户端失败", e);
        }
    }
}

改进

  • 性能优化：客户端实例缓存，避免重复创建
  • 配置管理：支持配置更新时刷新客户端
  • 线程安全：使用 ConcurrentHashMap

阶段六：完整的架构设计

最终架构

┌─────────────────────────────────────────────────────────┐
│                    Controller 层                        │
│          接收支付请求，调用 Service                      │
└────────────────────┬────────────────────────────────────┘
                     │
                     ↓
┌─────────────────────────────────────────────────────────┐
│                    Service 层                           │
│  PayOrderService: 处理支付订单业务逻辑                   │
│  └─→ PayChannelService: 获取支付渠道配置                │
└────────────────────┬────────────────────────────────────┘
                     │
                     ↓
┌─────────────────────────────────────────────────────────┐
│                    Factory 层                           │
│  PayClientFactory: 工厂模式 + 策略模式                  │
│  ├─ 策略映射表: channelCode → ClientClass              │
│  ├─ 客户端缓存: channelId → ClientInstance             │
│  └─ 动态创建/更新客户端                                  │

                     │
                     ↓
┌─────────────────────────────────────────────────────────┐
│               AbstractPayClient (模板方法)                │
│  ├─ unifiedOrder() [final] 模板方法                     │
│  │   ├─ validate() 统一验证                             │
│  │   ├─ doUnifiedOrder() [abstract] 子类实现           │
│  │   └─ exception handling 统一异常处理                 │
│  └─ 其他统一方法...                                     │
└────────────────────┬────────────────────────────────────┘
                     │
          ┌──────────┴──────────┐
          ↓                     ↓
┌─────────────────┐   ┌─────────────────┐
│ WxNativePayClient│   │AlipayWapPayClient│
│  doUnifiedOrder() │   │ doUnifiedOrder() │
└─────────────────┘   └─────────────────┘

完整代码示例

// ========== 1. 接口定义 ==========
public interface PayClient {
    String unifiedOrder(PayRequest request);
}

// ========== 2. 抽象类（模板方法） ==========
public abstract class AbstractPayClient implements PayClient {
    protected Long channelId;
    protected PayClientConfig config;

    public AbstractPayClient(Long channelId, PayClientConfig config) {
        this.channelId = channelId;
        this.config = config;
    }

    // 模板方法：统一流程
    @Override
    public final String unifiedOrder(PayRequest request) {
        validateRequest(request);  // 统一验证

        try {
            return doUnifiedOrder(request);  // 子类实现
        } catch (Exception e) {
            handleException(e);  // 统一异常处理
            throw new PayException("支付失败", e);
        }
    }

    protected abstract String doUnifiedOrder(PayRequest request);

    private void validateRequest(PayRequest request) {
        // 统一验证逻辑
    }

    private void handleException(Exception e) {
        // 统一异常处理
    }
}

// ========== 3. 具体实现（策略） ==========
public class WxNativePayClient extends AbstractPayClient {
    public WxNativePayClient(Long channelId, WxPayClientConfig config) {
        super(channelId, config);
    }

    @Override
    protected String doUnifiedOrder(PayRequest request) {
        // 微信Native支付逻辑
        return "wx_native_result";
    }
}

// ========== 4. 工厂（策略选择 + 实例管理） ==========
public class PayClientFactory {
    private final Map<String, Class<? extends AbstractPayClient>> strategyMap = new HashMap<>();
    private final Map<Long, AbstractPayClient> clientCache = new ConcurrentHashMap<>();

    {
        strategyMap.put("wx_native", WxNativePayClient.class);
        strategyMap.put("alipay_wap", AlipayWapPayClient.class);
    }

    public PayClient getOrCreate(Long channelId, String channelCode, PayClientConfig config) {
        AbstractPayClient client = clientCache.get(channelId);
        if (client == null) {
            Class<? extends AbstractPayClient> clientClass = strategyMap.get(channelCode);
            client = ReflectUtil.newInstance(clientClass, channelId, config);
            clientCache.put(channelId, client);
        }
        return client;
    }
}

// ========== 5. Service 使用 ==========
public class PaymentService {
    @Autowired
    private PayClientFactory factory;

    public String pay(Long channelId, String channelCode, PayRequest request) {
        // 从数据库获取配置
        PayClientConfig config = getConfigFromDB(channelId);

        // 工厂获取客户端（策略模式：根据channelCode选择策略）
        PayClient client = factory.getOrCreate(channelId, channelCode, config);

        // 调用支付（模板方法：统一流程）
        return client.unifiedOrder(request);
    }
}

设计思路总结

我的实现思路（从零开始）

  1. 第一步：定义接口
    • 思考：支付客户端需要哪些方法？
    • 结果：PayClient 接口
  2. 第二步：实现具体类
    • 思考：每个支付渠道怎么实现？
    • 结果：WxNativePayClient、AlipayWapPayClient 等
  3. 第三步：发现重复代码
    • 问题：每个实现类都有验证、异常处理等重复代码
    • 解决：提取抽象类 AbstractPayClient，使用模板方法模式
  4. 第四步：消除 if-else
    • 问题：创建客户端时有很多 if-else
    • 解决：用 Map 存储策略映射，引入工厂模式
  5. 第五步：优化性能
    • 问题：每次都创建新实例，性能差
    • 解决：工厂管理缓存，复用实例
  6. 第六步：支持配置更新
    • 问题：配置更新时需要刷新客户端
    • 解决：工厂支持检测配置变化并刷新

设计模式的应用时机

  • 模板方法模式：当发现多个类有相同的执行流程时
  • 策略模式：当发现有很多 if-else 需要选择不同实现时
  • 工厂模式：当需要统一管理对象创建和生命周期时

关键设计原则

  1. 单一职责：每个类只做一件事
  2. 开闭原则：对扩展开放，对修改关闭
  3. 依赖倒置：依赖抽象，不依赖具体实现
