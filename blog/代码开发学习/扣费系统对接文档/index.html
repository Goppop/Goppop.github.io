扣费系统对接文档

1. 概述

扣费系统为订单系统提供统一的扣费和退款能力，支持一次性扣费和周期性扣费两种模式。

核心能力：

  • 一次性扣费（包月预付费）
  • 周期扣费（按小时后付费）
  • 全额退款和部分退款
  • 停止周期扣费
  • 余额预警（提前3天和1天预警）
  • 幂等性保证

服务信息：

  • 服务名称：pay-server
  • 基础路径：/admin-api/pay/deduction

快速导航：

  • 如何对接扣费接口：见 第2章：快速开始
  • 如何使用余额预警：见 第7.4节：订单系统如何使用
  • 消息推送说明：见 第6章：消息推送
  • 典型场景示例：见 第5章：典型场景

3. 核心概念

3.1 扣费类型

类型 说明 是否周期
MONTHLY_PREPAID 包月预付费 否
HOURLY_POSTPAID 按小时后付费 是

3.2 主记录与明细记录

主记录（DeductionRecord）：

  • 每个订单只有1条主记录
  • 记录订单级别的扣费配置
  • 周期扣费时记录当前期数和下次扣费时间

明细记录（DeductionDetail）：

  • 每次实际扣费产生1条明细
  • 记录每期的扣费金额、状态、时间
  • 支持独立退款

关系说明：

  • 一次性扣费：1个主记录 + 1个明细记录
  • 周期扣费：1个主记录 + N个明细记录（N=实际扣费次数）

3.3 扣费状态

状态 说明
PENDING 待扣费
SUCCESS 扣费成功
FAILED 扣费失败
STOPPED 已停止（仅周期扣费）

4. API接口

4.1 执行扣费

接口路径： POST /admin-api/pay/deduction/api/deduct

请求参数：

参数名 类型 必填 说明
orderNo String 是 订单号，全局唯一
userId Long 是 用户ID
instanceId String 否 实例ID（算力订单必填）
deductionType String 是 扣费类型：MONTHLY_PREPAID/HOURLY_POSTPAID
billingType String 是 计费方式：MONTHLY/HOURLY
amount BigDecimal 是 扣费金额
unitPrice BigDecimal 否 单价（周期扣费时必填）
durationHours BigDecimal 否 时长（小时）
isPeriodic Boolean 否 是否周期扣费，默认false
periodHours Integer 否 扣费周期（小时），默认1
totalPeriods Integer 否 总扣费次数，-1表示无限

响应参数：

参数名 类型 说明
success Boolean 是否成功
deductionNo String 扣费单号
orderNo String 订单号
amount BigDecimal 扣费金额
deductionTime LocalDateTime 扣费时间
failReason String 失败原因

请求示例：

{
  "orderNo": "ORDER_20251210_001",
  "userId": 1,
  "instanceId": "INS_001",
  "deductionType": "HOURLY_POSTPAID",
  "billingType": "HOURLY",
  "amount": 10.00,
  "unitPrice": 10.00,
  "durationHours": 1,
  "isPeriodic": true,
  "periodHours": 1,
  "totalPeriods": -1
}

响应示例：

{
  "code": 0,
  "data": {
    "success": true,
    "deductionNo": "DED1998573950626086912",
    "orderNo": "ORDER_20251210_001",
    "amount": 10.00,
    "deductionTime": "2025-12-10T10:09:04"
  },
  "msg": "成功"
}

4.2 按订单号退款

接口路径： POST /admin-api/pay/deduction/api/refund-by-order

请求参数：

参数名 类型 必填 说明
orderNo String 是 订单号
reason String 是 退款原因

响应参数：

参数名 类型 说明
success Boolean 是否成功
orderNo String 订单号
refundAmount BigDecimal 退款金额
refundTime LocalDateTime 退款时间
failReason String 失败原因

说明：

  • 退还该订单下所有已扣费且未退款的金额
  • 周期扣费会自动停止
  • 支持重复调用（幂等）

4.3 按扣费单号退款

接口路径： POST /admin-api/pay/deduction/api/refund-by-deduction

请求参数：

参数名 类型 必填 说明
deductionNo String 是 扣费单号
reason String 是 退款原因

响应参数： 同4.2

说明：

  • 只退还指定扣费单的金额
  • 适用于部分订单退款场景

4.4 部分退款

接口路径： POST /admin-api/pay/deduction/api/partial-refund

请求参数：

参数名 类型 必填 说明
deductionNo String 是 扣费单号
refundAmount BigDecimal 是 退款金额
reason String 是 退款原因

响应参数： 同4.2

说明：

  • 只退款指定金额
  • 退款金额不能超过可退金额
  • 可退金额 = 已扣金额 - 已退金额

4.5 按订单号停止周期扣费

接口路径： POST /admin-api/pay/deduction/api/stop-periodic-by-order

请求参数：

参数名 类型 必填 说明
orderNo String 是 订单号

响应参数：

参数名 类型 说明
data Boolean 是否成功

说明：

  • 停止该订单的周期扣费
  • 不影响已扣费的金额
  • 非周期扣费调用会返回失败

4.6 按扣费单号停止周期扣费

接口路径： POST /admin-api/pay/deduction/api/stop-periodic-by-deduction

请求参数：

参数名 类型 必填 说明
deductionNo String 是 扣费单号

响应参数： 同4.5

5. 典型场景

5.1 场景1：用户购买包月算力

// 1. 创建订单
String orderNo = orderService.createOrder(userId, specId, 1, "MONTHLY");

// 2. 执行一次性扣费
DeductionRequestDTO request = new DeductionRequestDTO();
request.setOrderNo(orderNo);
request.setUserId(userId);
request.setInstanceId(instanceId);
request.setDeductionType("MONTHLY_PREPAID");
request.setBillingType("MONTHLY");
request.setAmount(new BigDecimal("100.00"));
request.setUnitPrice(new BigDecimal("100.00"));
request.setDurationHours(new BigDecimal("720")); // 30天
request.setIsPeriodic(false);

CommonResult<DeductionResultDTO> result = deductionApi.deduct(request);

// 3. 保存扣费单号
if (result.isSuccess() && result.getData().getSuccess()) {
    orderService.updateDeductionNo(orderNo, result.getData().getDeductionNo());
}

5.6 场景6：处理余额预警

/**
 * 余额预警消息消费者
 */
@Component
@Slf4j
public class BalanceWarningConsumer {
    
    @Resource
    private OrderService orderService;
    
    @Resource
    private NotificationService notificationService;
    
    @RabbitListener(queues = "deduction.balance.warning.queue")
    public void handleBalanceWarning(BalanceWarningMessage message) {
        log.info("[余额预警] orderNo: {}, warningLevel: {}", 
                message.getOrderNo(), message.getWarningLevel());
        
        // 1. 查询订单信息
        Order order = orderService.getOrderByNo(message.getOrderNo());
        
        // 2. 计算剩余可用时间
        long remainingHours = (long) message.getRemainingPeriods() * message.getPeriodHours();
        
        // 3. 根据预警级别发送不同通知
        if ("1day".equals(message.getWarningLevel())) {
            // 紧急预警：发送短信
            String content = String.format(
                "【紧急】您的订单%s余额即将耗尽（剩余%d小时），请尽快充值！",
                order.getOrderName(), remainingHours
            );
            notificationService.sendSms(message.getUserId(), content);
        } else {
            // 普通预警：发送站内信
            String content = String.format(
                "您的订单%s余额预计%d天后耗尽，建议及时充值。",
                order.getOrderName(), remainingHours / 24
            );
            notificationService.sendInApp(message.getUserId(), content);
        }
        
        // 4. 记录预警历史
        orderService.saveWarningHistory(message);
    }
}

6. 消息推送

扣费系统通过RabbitMQ推送以下消息，订单系统可订阅相关消息：

6.1 扣费结果消息

Exchange: pay.deduction RoutingKey: pay.deduction.result 消息体：

{
  "deductionNo": "DED1998573950626086912",
  "orderNo": "ORDER_20251210_001",
  "userId": 1,
  "amount": 10.00,
  "success": true,
  "failReason": null,
  "deductionTime": "2025-12-10T10:09:04"
}

6.2 余额预警消息

Exchange: pay.deduction RoutingKey: pay.deduction.balance.warning 消息体：

{
  "userId": 1,
  "orderNo": "ORDER_20251210_001",
  "instanceId": "INS_001",
  "currentBalance": 100.00,
  "unitPrice": 10.00,
  "remainingPeriods": 10,
  "periodHours": 1,
  "estimatedDepletionTime": "2025-12-12T10:00:00",
  "warningTime": "2025-12-09T10:00:00",
  "warningLevel": "3days"
}

预警级别说明：

  • 3days: 提前3天预警（剩余时间 ≤ 72小时）
  • 1day: 提前1天预警（剩余时间 ≤ 24小时）

预警机制：

  • 定时任务每小时检查一次所有活跃的周期扣费订单
  • 根据当前余额和扣费周期计算剩余可用时间
  • 达到预警阈值时发送预警消息（通过Redis去重，避免重复发送）

处理建议：

  • 发送站内信或短信通知用户充值
  • 根据预警级别设置不同的通知优先级（1天预警更紧急）
  • 记录预警历史，用于后续分析
  • 用户充值后预警自动解除

6.3 余额不足消息

Exchange: pay.deduction RoutingKey: pay.deduction.balance.insufficient 消息体：

{
  "deductionNo": "DED1998573950626086912",
  "orderNo": "ORDER_20251210_001",
  "userId": 1,
  "requiredAmount": 10.00,
  "currentBalance": 5.00,
  "instanceId": "INS_001",
  "checkTime": "2025-12-10T10:09:04"
}

处理建议：

  • 发送站内信或短信通知用户
  • 立即暂停或关闭用户实例
  • 记录欠费状态
  • 周期扣费已自动停止，无需再次调用停止接口

6.4 退款结果消息

Exchange: pay.deduction RoutingKey: pay.deduction.refund.result 消息体：

{
  "orderNo": "ORDER_20251210_001",
  "userId": 1,
  "refundAmount": 50.00,
  "success": true,
  "refundTime": "2025-12-10T10:09:04"
}

7. 余额预警系统

重要说明：余额预警系统是自动运行的，订单系统无需主动调用任何接口，只需要订阅消息队列接收预警通知即可。

7.1 预警机制说明

扣费系统内置了余额预警功能，会提前通知用户余额不足，避免服务突然中断。

两级预警策略：

预警级别 触发条件 预警时间 说明
一级预警 剩余时间 ≤ 72小时 提前3天 温馨提醒用户充值
二级预警 剩余时间 ≤ 24小时 提前1天 紧急通知用户充值

计算公式：

剩余可扣费次数 = 当前余额 / 单次扣费金额（向下取整）
剩余可用时间 = 剩余可扣费次数 × 扣费周期（小时）

示例：

  • 当前余额：100元
  • 每小时扣费：10元
  • 剩余可用时间：10小时
  • 触发级别：二级预警（1天预警）

7.2 预警触发方式

1. 定时扫描（主要方式）

  • 定时任务每小时扫描所有活跃的周期扣费订单
  • 计算每个订单的剩余可用时间
  • 达到预警阈值时自动发送预警消息

2. 实时检查（辅助方式）

  • 在执行周期扣费前实时检查余额
  • 如果剩余时间不足1天，立即发送预警
  • 确保及时通知用户

7.3 预警去重机制

Redis去重：

  • 使用Redis记录每个订单的预警状态
  • Redis Key格式：
    • 3天预警：balance:warning:3days:{orderNo}
    • 1天预警：balance:warning:1day:{orderNo}
  • 预警记录有效期：4天
  • 同一订单在4天内不会重复发送相同级别的预警

多级预警：

  • 3天预警和1天预警独立记录
  • 用户可能先收到3天预警，3天后再收到1天预警
  • 如果用户充值后余额恢复，预警会在4天后自动过期

7.4 订单系统如何使用

预警系统是自动运行的，订单系统无需主动调用，只需要订阅消息并处理即可。

使用流程

graph LR
    A[订单系统创建订单] --> B[调用扣费接口]
    B --> C[扣费系统自动监控]
    C --> D[定时任务检查余额]
    D --> E{余额不足?}
    E -->|是| F[发送预警消息]
    F --> G[订单系统接收消息]
    G --> H[通知用户充值]
    E -->|否| D

第1步：创建RabbitMQ消费者

在订单系统中创建消费者监听预警消息：

package cn.ksxzcx.cloud.module.order.mq.consumer;

import cn.ksxzcx.cloud.module.pay.mq.message.BalanceWarningMessage;
import lombok.extern.slf4j.Slf4j;
import org.springframework.amqp.rabbit.annotation.RabbitListener;
import org.springframework.stereotype.Component;

import javax.annotation.Resource;

/**
 * 余额预警消息消费者
 */
@Component
@Slf4j
public class BalanceWarningConsumer {
    
    @Resource
    private OrderService orderService;
    
    @Resource
    private NotificationService notificationService;
    
    @Resource
    private SmsService smsService;
    
    /**
     * 处理余额预警消息
     */
    @RabbitListener(queues = "deduction.balance.warning.queue")
    public void handleBalanceWarning(BalanceWarningMessage message) {
        log.info("[余额预警] 用户: {}, 订单: {}, 预警级别: {}, 剩余可用时长: {}小时", 
                message.getUserId(), 
                message.getOrderNo(), 
                message.getWarningLevel(),
                message.getRemainingPeriods() * message.getPeriodHours());
        
        try {
            // 1. 查询订单信息
            Order order = orderService.getOrderByNo(message.getOrderNo());
            if (order == null) {
                log.warn("[余额预警][订单不存在] orderNo: {}", message.getOrderNo());
                return;
            }
            
            // 2. 根据预警级别发送不同通知
            if ("1day".equals(message.getWarningLevel())) {
                // 1天预警：更紧急，发送短信+站内信
                sendUrgentNotification(message, order);
            } else if ("3days".equals(message.getWarningLevel())) {
                // 3天预警：发送站内信+邮件
                sendNormalNotification(message, order);
            }
            
            // 3. 记录预警历史（用于后续分析）
            orderService.saveWarningHistory(message);
            
            log.info("[余额预警][处理完成] orderNo: {}", message.getOrderNo());
            
        } catch (Exception e) {
            log.error("[余额预警][处理失败] orderNo: {}, userId: {}", 
                    message.getOrderNo(), message.getUserId(), e);
        }
    }
    
    /**
     * 发送紧急通知（1天预警）
     */
    private void sendUrgentNotification(BalanceWarningMessage message, Order order) {
        // 计算剩余时间
        long remainingHours = (long) message.getRemainingPeriods() * message.getPeriodHours();
        
        // 1. 发送短信
        String smsContent = String.format(
            "【算力平台】尊敬的用户，您的订单%s余额即将耗尽（预计剩余%d小时），请及时充值以免服务中断。",
            order.getOrderName(),
            remainingHours
        );
        smsService.sendSms(message.getUserId(), smsContent);
        
        // 2. 发送站内信
        String notificationContent = String.format(
            "您的订单【%s】余额预警\n" +
            "当前余额：%.2f元\n" +
            "预计剩余时长：%d小时\n" +
            "预计耗尽时间：%s\n" +
            "为避免服务中断，请及时充值。",
            order.getOrderName(),
            message.getCurrentBalance(),
            remainingHours,
            message.getEstimatedDepletionTime()
        );
        notificationService.sendWithAction(
            message.getUserId(), 
            "余额预警", 
            notificationContent,
            "立即充值",
            "/recharge"
        );
    }
    
    /**
     * 发送普通通知（3天预警）
     */
    private void sendNormalNotification(BalanceWarningMessage message, Order order) {
        long remainingHours = (long) message.getRemainingPeriods() * message.getPeriodHours();
        
        String content = String.format(
            "您的订单【%s】余额提醒\n" +
            "当前余额：%.2f元\n" +
            "预计剩余时长：%d小时（约%d天）\n" +
            "建议您及时充值，避免服务中断。",
            order.getOrderName(),
            message.getCurrentBalance(),
            remainingHours,
            remainingHours / 24
        );
        
        notificationService.send(message.getUserId(), "余额提醒", content);
    }
}

方式2：监听余额不足消息

余额不足导致扣费失败时，也会停止周期扣费并发送余额不足消息（见6.3节）

7.5 预警消息字段说明

字段 类型 必填 说明 示例
userId Long 是 用户ID 1
orderNo String 是 订单号 ORDER_20251210_001
instanceId String 否 实例ID（算力订单） INS_001
currentBalance BigDecimal 是 当前余额（元） 100.00
unitPrice BigDecimal 是 单次扣费金额（元） 10.00
remainingPeriods Integer 是 剩余可扣费次数 10
periodHours Integer 是 扣费周期（小时） 1
estimatedDepletionTime LocalDateTime 是 预计余额耗尽时间 2025-12-12T10:00:00
warningTime LocalDateTime 是 预警时间 2025-12-09T10:00:00
warningLevel String 是 预警级别 “3days” 或 “1day”

字段说明：

  1. remainingPeriods：剩余可扣费次数 = 当前余额 / 单次扣费金额（向下取整）
  2. estimatedDepletionTime：预计余额耗尽时间 = 当前时间 + (剩余次数 × 扣费周期)
  3. warningLevel：
    • "3days"：剩余时间 ≤ 72小时
    • "1day"：剩余时间 ≤ 24小时

计算示例：

当前余额：100元
单次扣费：10元/小时
扣费周期：1小时

剩余可扣费次数 = 100 / 10 = 10次
剩余可用时间 = 10 × 1 = 10小时
预计耗尽时间 = 当前时间 + 10小时
预警级别 = "1day"（因为10小时 < 24小时）

7.7 重要说明

1. 预警系统完全自动运行

  • 订单系统无需主动调用任何预警接口
  • 只需订阅消息队列即可接收预警
  • 扣费系统会自动监控并发送预警

2. 预警不会重复发送

  • 通过Redis确保同一订单4天内不会重复发送相同级别的预警
  • 用户充值后，预警标记会在4天后自动过期
  • 如果余额再次不足，会重新触发预警

3. 预警与扣费失败的区别

  • 预警：余额充足，但即将不足（提前通知）
  • 扣费失败：余额不足，扣费已经失败（事后通知）
  • 建议同时监听两种消息，提供最佳用户体验

4. 预警消息包含的关键信息

  • 当前余额
  • 剩余可扣费次数
  • 预计余额耗尽时间
  • 预警级别（3天/1天）

7.9 常见问题

Q1：预警消息会重复发送吗？ A：不会。通过Redis确保同一订单在4天内不会重复发送相同级别的预警。

Q2：用户充值后预警会立即解除吗？ A：预警标记会在4天后自动过期。如果充值后余额充足，下次定时任务检查时不会再触发预警。

Q3：为什么同时有定时扫描和实时检查？ A：

  • 定时扫描：确保所有订单都会被检查到
  • 实时检查：在扣费时立即发现余额不足，更及时

Q4：预警消息发送失败怎么办？ A：预警失败不影响扣费流程。下次定时任务会重新检查，如果仍然满足条件且未发送过，会再次尝试发送。

Q5：可以自定义预警阈值吗？ A：当前固定为3天和1天。如需自定义，可以在 BalanceWarningService 中修改常量。

Q6：预警消息的优先级如何？ A：1天预警优先级高于3天预警。如果同时满足两个条件，会优先发送1天预警。

8. 重要说明

8.1 幂等性保证

所有扣费和退款接口均保证幂等性：

  • 相同订单号重复调用扣费，只会扣费一次
  • 相同订单号重复调用退款，只会退款一次
  • 建议在业务系统中也保存扣费单号，用于对账

8.2 周期扣费说明

首次扣费：

  • 调用扣费接口时立即扣费第1期
  • 扣费成功后，系统会自动创建周期任务

续扣机制：

  • 定时任务每分钟扫描需要续扣的订单
  • 自动扣减余额并创建扣费明细
  • 余额不足时立即停止周期扣费

停止扣费：

  • 用户释放实例时必须调用停止接口
  • 退款时会自动停止周期扣费
  • 余额不足时立即自动停止

8.3 金额说明

  • 所有金额单位：元
  • 金额精度：小数点后4位（数据库DECIMAL(18,4)）
  • 可退金额 = 已扣金额 - 已退金额
  • 超过4位小数的金额会被四舍五入
