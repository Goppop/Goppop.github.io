Redis使用总结

代码开发学习Redis
PublishedApr 14, 2025

第一部分：Redis常用场景

1. 缓存存储（最常用）

场景1：OAuth2 Token缓存

需求： 用户登录后，将access token缓存到Redis，避免每次都查数据库

实现： OAuth2AccessTokenRedisDAO.java

@Repository
public class OAuth2AccessTokenRedisDAO {

    @Resource
    private StringRedisTemplate stringRedisTemplate;

    /**
     * 获取Token
     */
    public OAuth2AccessTokenDO get(String accessToken) {
        String redisKey = formatKey(accessToken);
        String value = stringRedisTemplate.opsForValue().get(redisKey);
        // JSON字符串转对象
        return JsonUtils.parseObject(value, OAuth2AccessTokenDO.class);
    }

    /**
     * 设置Token（带过期时间）
     */
    public void set(OAuth2AccessTokenDO accessTokenDO) {
        String redisKey = formatKey(accessTokenDO.getAccessToken());
        // 计算过期时间
        long expireSeconds = LocalDateTimeUtil.between(
            LocalDateTime.now(),
            accessTokenDO.getExpiresTime(),
            ChronoUnit.SECONDS
        );
        if (expireSeconds > 0) {
            // 设置缓存，带过期时间
            stringRedisTemplate.opsForValue().set(
                redisKey,
                JsonUtils.toJsonString(accessTokenDO),
                expireSeconds,
                TimeUnit.SECONDS
            );
        }
    }

    /**
     * 删除Token
     */
    public void delete(String accessToken) {
        String redisKey = formatKey(accessToken);
        stringRedisTemplate.delete(redisKey);
    }
}

关键点：

  • opsForValue()：操作String类型
  • set(key, value, timeout, timeUnit)：设置带过期时间
  • 使用JSON序列化存储对象

场景2：验证码缓存

需求： 验证码需要临时存储，5分钟后过期

public class RedisCaptchaServiceImpl implements CaptchaCacheService {

  • 短信验证码：3分钟过期
  • 邮箱验证码：10分钟过期

2. 原子操作（保证并发安全）

场景4：幂等性保证（setIfAbsent）

需求： 防止重复请求，保证接口幂等性

实现： IdempotentRedisDAO.java

public class IdempotentRedisDAO {

    private final StringRedisTemplate redisTemplate;

    /**
     * 尝试设置Key（如果不存在）
     * 相当于分布式锁的简单实现
     */
    public Boolean setIfAbsent(String key, long timeout, TimeUnit timeUnit) {
        String redisKey = formatKey(key);
        // setIfAbsent：如果key不存在才设置，返回true
        // 如果key已存在，返回false（说明是重复请求）
        return redisTemplate.opsForValue().setIfAbsent(redisKey, "", timeout, timeUnit);
    }

    /**
     * 删除Key
     */
    public void delete(String key) {
        String redisKey = formatKey(key);
        redisTemplate.delete(redisKey);
    }
}

使用方式：

// 在切面中使用
@Around("@annotation(idempotent)")
public Object aroundPointCut(ProceedingJoinPoint joinPoint, Idempotent idempotent) {
    String key = keyResolver.resolver(joinPoint, idempotent);

    // 尝试设置Key（原子操作）
    boolean success = idempotentRedisDAO.setIfAbsent(key, timeout, timeUnit);
    if (!success) {
        // 重复请求，抛出异常
        throw new ServiceException(REPEATED_REQUESTS);
    }

    try {
        return joinPoint.proceed();
    } catch (Throwable e) {
        // 异常时删除Key
        idempotentRedisDAO.delete(key);
        throw e;
    }
}

关键点：

  • setIfAbsent：原子操作，保证并发安全
  • 设置过期时间，避免Key永久存在
  • 异常时删除Key，允许重试

场景5：防止重复下单（setIfAbsent）

需求： 防止用户同时创建多个未支付的充值订单

实现： AccountRechargeRedisDAO.java

@Repository
public class AccountRechargeRedisDAO {

    @Resource
    private StringRedisTemplate stringRedisTemplate;

    private static final long TIMEOUT_MINUTES = 30;  // 30分钟过期

    /**
     * 检查用户是否有未支付的充值订单
     */
    public String getPendingRecharge(Long userId) {
        String key = formatKey(userId);
        return stringRedisTemplate.opsForValue().get(key);
    }

    /**
     * 设置用户未支付的充值订单（原子操作）
     */
    public Boolean setPendingRecharge(Long userId, String rechargeNo) {
        String key = formatKey(userId);
        // 使用setIfAbsent保证原子性
        // 如果用户已有未支付订单，返回false
        Boolean success = stringRedisTemplate.opsForValue()
                .setIfAbsent(key, rechargeNo, Duration.ofMinutes(TIMEOUT_MINUTES));

        if (Boolean.TRUE.equals(success)) {
            log.info("[setPendingRecharge][设置成功] userId: {}, rechargeNo: {}", userId, rechargeNo);
        } else {
            log.warn("[setPendingRecharge][用户已存在未支付订单] userId: {}", userId);
        }
        return success;
    }

    /**
     * 删除用户未支付的充值订单（支付成功后）
     */
    public void deletePendingRecharge(Long userId) {
        String key = formatKey(userId);
        stringRedisTemplate.delete(key);
    }

    /**
     * 获取订单剩余有效时间（秒）
     */
    public Long getExpireTime(Long userId) {
        String key = formatKey(userId);
        return stringRedisTemplate.getExpire(key);  // 返回剩余秒数
    }
}

使用场景：

@Service
public class AccountRechargeService {

    @Resource
    private AccountRechargeRedisDAO rechargeRedisDAO;

    public Long createRecharge(Long userId, BigDecimal amount) {
        // 检查是否有未支付订单
        String existingOrder = rechargeRedisDAO.getPendingRecharge(userId);
        if (existingOrder != null) {
            throw exception(REPEATED_RECHARGE_REQUEST);
        }

        // 创建订单
        AccountRechargeDO recharge = new AccountRechargeDO();
        // ... 设置订单信息

        // 设置到Redis（原子操作，防止并发）
        Boolean success = rechargeRedisDAO.setPendingRecharge(userId, recharge.getRechargeNo());
        if (!success) {
            // 并发情况下，另一个请求已经设置了
            throw exception(REPEATED_RECHARGE_REQUEST);
        }

        // 保存到数据库
        rechargeMapper.insert(recharge);
        return recharge.getId();
    }

    public void onPaySuccess(Long rechargeId) {
        // 支付成功后，删除Redis中的订单记录
        AccountRechargeDO recharge = rechargeMapper.selectById(rechargeId);
        rechargeRedisDAO.deletePendingRecharge(recharge.getUserId());
    }
}

关键点：

  • setIfAbsent：保证并发安全
  • 设置过期时间：30分钟自动过期
  • 支付成功后删除：释放限制

3. 计数器（incr）

场景6：验证码发送次数限制

需求： 限制用户每天发送验证码的次数

@Repository
public class SmsCodeRedisDAO {

    @Resource
    private StringRedisTemplate stringRedisTemplate;

    /**
     * 增加发送次数
     */
    public Long increment(String key) {
        // 增加1，如果key不存在，从0开始
        Long count = stringRedisTemplate.opsForValue().increment(key);
        // 设置过期时间（24小时）
        if (count == 1) {
            stringRedisTemplate.expire(key, 24, TimeUnit.HOURS);
        }
        return count;
    }

    /**
     * 检查是否超过限制
     */
    public boolean isExceedLimit(String key, int maxCount) {
        String countStr = stringRedisTemplate.opsForValue().get(key);
        if (countStr == null) {
            return false;  // 还没发送过
        }
        int count = Integer.parseInt(countStr);
        return count >= maxCount;
    }
}

使用方式：

public void sendSmsCode(String phone) {
    String key = "sms:code:" + phone;

    // 检查是否超过限制（每天最多5次）
    if (smsCodeRedisDAO.isExceedLimit(key, 5)) {
        throw exception(SMS_CODE_SEND_LIMIT_EXCEEDED);
    }

    // 发送验证码
    smsService.sendCode(phone);

    // 增加计数
    smsCodeRedisDAO.increment(key);
}

4. 限流（RateLimiter）

场景7：接口限流

需求： 限制接口的访问频率，防止被刷

实现： RateLimiterRedisDAO.java

public class RateLimiterRedisDAO {

    private final RedissonClient redissonClient;

    /**
     * 尝试获取限流令牌
     * @param key 限流Key
     * @param count 时间窗口内允许的次数
     * @param time 时间窗口
     * @param timeUnit 时间单位
     */
    public Boolean tryAcquire(String key, int count, int time, TimeUnit timeUnit) {
        // 1. 获取限流器
        RRateLimiter rateLimiter = getRRateLimiter(key, count, time, timeUnit);
        // 2. 尝试获取1个令牌
        return rateLimiter.tryAcquire();
    }

    private RRateLimiter getRRateLimiter(String key, long count, int time, TimeUnit timeUnit) {
        String redisKey = formatKey(key);
        RRateLimiter rateLimiter = redissonClient.getRateLimiter(redisKey);

        // 设置限流规则：时间窗口内允许count次
        long rateInterval = timeUnit.toSeconds(time);
        Duration duration = Duration.ofSeconds(rateInterval);
        rateLimiter.trySetRate(RateType.OVERALL, count, duration);
        rateLimiter.expire(duration);

        return rateLimiter;
    }
}

使用方式：

@Aspect
public class RateLimiterAspect {

    @Before("@annotation(rateLimiter)")
    public void beforePointCut(JoinPoint joinPoint, RateLimiter rateLimiter) {
        String key = keyResolver.resolver(joinPoint, rateLimiter);

        // 尝试获取限流令牌
        boolean success = rateLimiterRedisDAO.tryAcquire(
            key,
            rateLimiter.count(),  // 60秒内
            rateLimiter.time(),    // 10次
            rateLimiter.timeUnit()
        );

        if (!success) {
            // 超过限流，抛出异常
            throw new ServiceException(TOO_MANY_REQUESTS);
        }
    }
}

使用示例：

@RestController
public class ApiController {

    @RateLimiter(count = 10, time = 60)  // 60秒内最多10次
    @GetMapping("/api")
    public Result api() {
        // 业务逻辑
    }
}

第二部分：分布式锁常用场景

1. 钱包余额更新（防止并发）

场景1：钱包扣款（分布式锁）

需求： 多个请求同时扣款时，保证余额计算的准确性

实现： PayWalletLockRedisDAO.java

@Repository
public class PayWalletLockRedisDAO {

    @Resource
    private RedissonClient redissonClient;

    /**
     * 加锁执行逻辑
     * @param id 钱包ID
     * @param timeoutMillis 锁超时时间（毫秒）
     * @param callable 需要执行的逻辑
     */
    public <V> V lock(Long id, Long timeoutMillis, Callable<V> callable) throws Exception {
        String lockKey = formatKey(id);
        RLock lock = redissonClient.getLock(lockKey);
        try {
            // 加锁，设置超时时间
            lock.lock(timeoutMillis, TimeUnit.MILLISECONDS);
            // 执行逻辑
            return callable.call();
        } finally {
            // 释放锁
            lock.unlock();
        }
    }
}

使用方式： PayWalletServiceImpl.java

@Service
public class PayWalletServiceImpl {

    @Resource
    private PayWalletLockRedisDAO lockRedisDAO;

    private static final long UPDATE_TIMEOUT_MILLIS = 5000;  // 5秒超时

    /**
     * 扣减钱包余额
     */
    public void reduceWalletBalance(Long walletId, BigDecimal price) {
        // 加锁，保证并发安全
        lockRedisDAO.lock(walletId, UPDATE_TIMEOUT_MILLIS, () -> {
            // 1. 重新获取最新的钱包信息
            PayWalletDO wallet = getWallet(walletId);

            // 2. 检查余额是否充足
            if (wallet.getBalance().compareTo(price) < 0) {
                throw exception(WALLET_BALANCE_NOT_ENOUGH);
            }

            // 3. 扣除余额
            walletMapper.updateBalance(walletId, price.negate());

            // 4. 记录流水
            walletTransactionMapper.insert(new PayWalletTransactionDO()
                .setWalletId(walletId)
                .setAmount(price.negate())
                .setType(TransactionTypeEnum.DEDUCT.getType()));

            return null;
        });
    }
}

为什么需要锁？

并发场景：
请求1：读取余额 100元
请求2：读取余额 100元
请求1：扣除 50元，余额 = 50元
请求2：扣除 50元，余额 = 50元  ❌ 应该是0元

使用锁后：
请求1：加锁 → 读取余额 100元 → 扣除 50元 → 余额 = 50元 → 释放锁
请求2：等待锁 → 加锁 → 读取余额 50元 → 扣除 50元 → 余额 = 0元 → 释放锁 ✅

关键点：

  • 锁的粒度：按钱包ID加锁，不同钱包可以并发
  • 超时时间：防止死锁，设置合理的超时时间
  • 重新查询：加锁后重新查询，确保数据最新

2. 支付通知（防止重复通知）

场景2：支付通知去重

需求： 支付回调可能重复，需要保证只处理一次

实现： PayNotifyLockRedisDAO.java

@Repository
public class PayNotifyLockRedisDAO {

    @Resource
    private RedissonClient redissonClient;

    /**
     * 加锁执行通知逻辑
     */
    public void lock(Long id, Long timeoutMillis, Runnable runnable) {
        String lockKey = formatKey(id);
        RLock lock = redissonClient.getLock(lockKey);
        try {
            // 加锁
            lock.lock(timeoutMillis, TimeUnit.MILLISECONDS);
            // 执行通知逻辑
            runnable.run();
        } finally {
            // 释放锁
            lock.unlock();
        }
    }
}

使用方式： PayNotifyServiceImpl.java

@Service
public class PayNotifyServiceImpl {

    @Resource
    private PayNotifyLockRedisDAO notifyLockRedisDAO;

    private static final long NOTIFY_TIMEOUT_MILLIS = 120 * 1000;  // 2分钟超时

    /**
     * 执行支付通知
     */
    public void executeNotify(PayNotifyTaskDO task) {
        // 分布式锁，避免并发问题
        notifyLockRedisDAO.lock(task.getId(), NOTIFY_TIMEOUT_MILLIS, () -> {
            // 1. 重新查询任务，检查通知次数
            PayNotifyTaskDO dbTask = notifyTaskMapper.selectById(task.getId());
            if (ObjectUtil.notEqual(task.getNotifyTimes(), dbTask.getNotifyTimes())) {
                // 通知次数不匹配，说明已经被其他线程处理过了
                log.warn("[executeNotify][任务已被处理] taskId: {}", task.getId());
                return;
            }

            // 2. 执行通知
            getSelf().executeNotify0(dbTask);
        });
    }
}

为什么需要锁？

并发场景：
回调1：接收到支付成功回调
回调2：接收到支付成功回调（重复）
回调1：处理通知 → 更新订单状态
回调2：处理通知 → 重复更新订单状态 ❌

使用锁后：
回调1：加锁 → 处理通知 → 更新订单状态 → 释放锁
回调2：等待锁 → 加锁 → 检查已处理 → 直接返回 ✅

关键点：

  • 双重检查：加锁后再次检查，避免重复处理
  • 超时时间：设置合理的超时时间（2分钟）
  • 幂等性：即使重复处理，也要保证结果一致

3. 定时任务（防止多实例重复执行）

场景3：定时任务去重

需求： 多个服务实例同时运行，定时任务只执行一次

实现： RedisStreamMessageCleanupJob.java

@Slf4j
public class RedisStreamMessageCleanupJob {

    private static final String LOCK_KEY = "redis:stream:message-cleanup:lock";

    @Resource
    private RedissonClient redissonClient;

    /**
     * 每小时执行一次清理任务
     */
    @Scheduled(cron = "0 0 * * * ?")
    public void cleanup() {
        RLock lock = redissonClient.getLock(LOCK_KEY);
        // 尝试加锁（非阻塞）
        if (lock.tryLock()) {
            try {
                // 执行清理逻辑
                execute();
            } catch (Exception ex) {
                log.error("[cleanup][执行异常]", ex);
            } finally {
                lock.unlock();
            }
        } else {
            // 其他实例正在执行，直接返回
            log.debug("[cleanup][其他实例正在执行，跳过]");
        }
    }
}

关键点：

  • tryLock()：非阻塞，获取不到锁直接返回
  • 多个实例：只有一个实例能执行任务
  • 避免重复：防止多个实例同时执行定时任务

第三部分：Redis和分布式锁的区别

Redis原子操作 vs 分布式锁

场景对比

场景 使用方式 适用条件
简单幂等性 setIfAbsent 只需要判断是否存在，不需要执行复杂逻辑
复杂业务逻辑 分布式锁 需要执行多步操作，需要保证原子性

示例对比

使用setIfAbsent（简单）：

// 幂等性检查：只需要判断是否为重复请求
boolean success = redisTemplate.opsForValue().setIfAbsent(key, "", timeout, timeUnit);
if (!success) {
    throw exception(REPEATED_REQUESTS);  // 重复请求，直接返回
}
// 执行逻辑
doSomething();

使用分布式锁（复杂）：

// 钱包扣款：需要执行多步操作
lock.lock();
try {
    // 1. 查询余额
    WalletDO wallet = getWallet(walletId);
    // 2. 检查余额
    if (wallet.getBalance() < price) {
        throw exception(BALANCE_NOT_ENOUGH);
    }
    // 3. 扣除余额
    updateBalance(walletId, price);
    // 4. 记录流水
    insertTransaction(...);
} finally {
    lock.unlock();
}

Redis使用场景

“在项目中，Redis主要用于以下几个方面：

1. 缓存存储：

  • OAuth2 Token缓存：用户登录后，将access token缓存到Redis，设置过期时间，避免频繁查数据库
  • 验证码缓存：图形验证码、短信验证码缓存，设置5-10分钟过期

2. 原子操作：

  • 幂等性保证：使用setIfAbsent，防止重复请求。如果key已存在，说明是重复请求，直接返回
  • 防止重复下单：使用setIfAbsent，保证用户同时只能有一个未支付订单

3. 限流：

  • 接口限流：使用Redisson的RRateLimiter，限制接口访问频率，比如60秒内最多10次请求

4. 计数器：

  • 验证码发送次数：使用increment，限制用户每天发送验证码的次数“

分布式锁使用场景（1-2分钟）

“在项目中，分布式锁主要用于：

1. 钱包余额更新：

  • 多个请求同时扣款时，使用分布式锁保证余额计算的准确性
  • 按钱包ID加锁，不同钱包可以并发操作
  • 加锁后重新查询余额，确保数据最新

2. 支付通知去重：

  • 支付回调可能重复，使用分布式锁保证只处理一次
  • 加锁后再次检查通知次数，避免重复处理

3. 定时任务去重：

  • 多个服务实例同时运行，使用tryLock()保证定时任务只执行一次

技术选型：

  • 使用Redisson实现分布式锁
  • 支持自动续期，防止业务执行时间过长导致锁过期
  • 使用tryLock()非阻塞方式，避免线程阻塞“

总结

Redis常用操作

  1. String操作
    • set(key, value, timeout, timeUnit)：设置带过期时间
    • get(key)：获取值
    • setIfAbsent(key, value, timeout, timeUnit)：原子操作
    • delete(key)：删除
    • increment(key)：自增
  2. 应用场景
    • 缓存：Token、验证码、会话
    • 原子操作：幂等性、防止重复
    • 限流：接口限流
    • 计数器：发送次数限制

分布式锁常用操作

  1. Redisson锁
    • lock.lock(timeout, timeUnit)：加锁
    • lock.tryLock()：非阻塞加锁
    • lock.unlock()：释放锁
  2. 应用场景
    • 并发控制：钱包扣款、余额更新
    • 去重处理：支付通知、定时任务
    • 资源保护：防止并发操作

选择原则

  • 简单场景：用Redis原子操作（setIfAbsent）
  • 复杂场景：用分布式锁（需要执行多步操作）
