MySQL 数据库优化专题：从慢 SQL 到 InnoDB 内核

摘要

数据库优化并非单一参数调节或索引补充问题，而是一个由负载观测、执行计划解释、存储引擎机制、事务并发控制和架构治理共同构成的系统性工程问题。对于应用系统而言，最直接的优化入口通常来自线上慢 SQL：阿里云 RDS / PolarDB 的 SQL 洞察、慢 SQL 统计和性能趋势图能够提供执行次数、平均耗时、最大耗时、扫描行数、返回行数、锁等待、CPU 与 IO 趋势等基础证据。

然而，监控系统只能指出“哪些 SQL 出现异常”，并不能直接回答“异常为何产生”以及“何种修改具有最小风险和最大收益”。因此，数据库优化必须建立从现象采集到机制解释、从局部改写到系统验证的分析链条。

本文以 MySQL 8.0+ 与 InnoDB 存储引擎为主要对象，围绕慢 SQL 识别、执行计划分析、索引设计、Buffer Pool、Redo / Undo Log、MVCC、锁机制、大表治理和在线 DDL 等主题展开论述。文章的目的不是罗列优化技巧，而是建立一套可复用的数据库性能分析框架。

引言：研究对象与分析路径

本文讨论的核心问题可以表述为：

当业务系统通过阿里云 SQL 监控发现慢 SQL 后，如何基于执行计划和 InnoDB 内部机制判断瓶颈来源，并选择可验证、可回滚、可持续治理的优化方案？

为回答该问题，全文按以下路径展开：

慢 SQL 监控
  → 执行计划分析
  → 索引与 SQL 改写
  → InnoDB 存储与日志机制
  → 事务隔离与锁等待
  → 大表治理与架构演进
  → 诊断 SQL 与决策树

需要说明的是，不同 MySQL 版本、云数据库产品形态、参数配置和业务负载会影响具体结论。本文给出的是分析框架和工程推理路径，而不是可无条件套用的配置模板。

1. 从阿里云慢 SQL 监控开始

在真实生产环境中，数据库性能问题通常首先表现为接口响应时间上升、慢查询告警增加、CPU 利用率异常、连接池耗尽或锁等待堆积。阿里云 SQL 监控的价值在于将这些现象具象化为可分析的 SQL 样本和实例级指标。

基于慢 SQL 样本的优化流程可以抽象为：

SQL 监控发现慢 SQL
    ↓
确认影响：执行次数、平均耗时、最大耗时、扫描行数、返回行数
    ↓
拿到 SQL、表结构、索引、数据量、查询参数分布
    ↓
EXPLAIN / EXPLAIN ANALYZE 分析执行计划
    ↓
判断瓶颈：索引、排序、临时表、锁等待、IO、返回数据量
    ↓
修改 SQL / 索引 / 表结构 / 访问模式
    ↓
测试环境验证执行计划和耗时
    ↓
灰度上线，继续观察慢 SQL、CPU、IO、锁等待、Buffer Pool

这一流程强调两个原则。第一，优化动作必须由证据驱动，而不是由经验规则直接驱动。第二，任何修改都必须经过执行计划、耗时和资源指标的复核。相同的慢 SQL 现象可能由不同原因触发：缺失索引、联合索引顺序不合理、数据量超过热数据边界、锁等待、统计信息失真、Buffer Pool 被大范围扫描污染，或者应用侧调用频率过高。

2. SQL 延迟的构成

SQL 延迟不是单点因素，而是多个阶段成本的叠加。一个查询请求从应用发出到结果返回，通常包含以下组成部分：

SQL 总耗时 =
  网络传输
+ 连接排队
+ 解析与优化
+ 锁等待
+ 数据读取
+ 过滤与计算
+ 排序 / 分组 / 临时表
+ 结果返回

不同成本项对应的优化策略并不相同：

慢的原因 常见表现 优先处理方向
扫描行太多 rows 很大，返回行很少 建索引、改写条件、减少扫描范围
没走合适索引 type=ALL 或 index 调整索引顺序、避免索引失效
排序成本高 Using filesort 让索引同时服务过滤和排序
临时表成本高 Using temporary 减少分组数据量，优化 GROUP BY
锁等待 SQL 本身不复杂但耗时长 查事务、锁等待、死锁、长事务
返回数据过多 扫描和返回都很大 分页、字段裁剪、冷热拆分
IO 压力大 磁盘读写高，Buffer Pool 命中率低 扩内存、减少扫描、归档冷数据

因此，慢 SQL 分析的第一步不是添加索引，而是判定主要瓶颈类型。索引只能解决访问路径问题，不能直接解决锁等待、连接池堆积、返回数据过大或实例级 IO 饱和等问题。

3. 慢 SQL 分析闭环

3.1 采集

慢 SQL 的证据来源通常包括三类：

  • 慢查询日志：适合离线分析 Top SQL。
  • performance_schema / sys：适合看当前实例运行状态。
  • APM / 业务埋点：适合把 SQL 耗时和接口耗时关联起来。

慢查询日志或云监控仅提供问题样本；根因判断还依赖表结构、索引定义、数据分布、执行计划、运行时锁状态以及实例资源指标。孤立地观察 SQL 文本，通常无法得到可靠结论。

3.2 分析

对单条 SQL，最常用的是：

EXPLAIN SELECT ...

MySQL 8.0 以后还可以使用：

EXPLAIN ANALYZE SELECT ...

EXPLAIN ANALYZE 会实际执行 SQL，并输出执行过程中每个节点的实际耗时和行数。它更接近真实情况，但因为会执行 SQL，生产环境要谨慎使用，尤其是写语句或大查询。

3.3 判断

阅读执行计划的重点不在于机械记忆字段含义，而在于回答以下问题：

  • 驱动表选得对不对？
  • 访问类型是全表扫描、索引扫描，还是范围扫描？
  • 预估扫描行数和实际返回行数差距大不大？
  • 是否出现临时表或文件排序？
  • 联合索引是否同时服务了过滤、排序和分组？
  • 是否有隐式类型转换、函数计算、前导模糊匹配导致索引失效？

3.4 修改

优化动作可以按照工程成本和风险从低到高排列：

  1. 改 SQL 写法：减少 SELECT *、避免函数包裹索引列、拆分复杂查询。
  2. 加或调整索引：联合索引顺序、覆盖索引、删除冗余索引。
  3. 改访问模式：游标分页、缓存、异步化、预聚合。
  4. 改表结构：字段拆分、冷热分离、分区、归档。
  5. 改架构：读写分离、分库分表、搜索引擎、OLAP 系统。

越靠后的方案，对系统结构的影响越大，也越需要灰度、回滚和监控验证。对于线上数据库而言，“可恢复性”与“可验证性”应当与性能收益同等重要。

4. 索引设计的基本原则

索引是数据库以空间成本和写入维护成本换取读取效率的结构化冗余。索引并非越多越好。过量索引会增加写入、更新和删除成本，占用 Buffer Pool，并可能使优化器选择空间变得更复杂。

4.1 B+Tree 为什么适合 MySQL

InnoDB 的主流索引结构是 B+Tree。其适用性来自磁盘数据库的基本约束：随机 IO 成本远高于内存访问成本，而 B+Tree 通过多路平衡结构显著降低树高，从而减少访问路径上的页读取次数。

在默认 16KB 页大小下，一个非叶子页可以容纳大量索引项。实际查询过程中，根节点和中间层索引页通常具有较高缓存命中率，磁盘访问压力主要集中在叶子页和回表读取上。

B+Tree 的结构特征直接决定了多项索引设计规则：

  • 非叶子节点只存 key 和指针，树高低。
  • 叶子节点按 key 有序，适合范围查询。
  • 叶子节点之间有链表，适合顺序扫描。
  • 索引列顺序决定了能否利用最左前缀。

4.2 联合索引的设计原则

联合索引设计时，可以按这个顺序思考：

  1. 等值条件优先放前面。
  2. 区分度高的列通常更适合靠前，但不是绝对。
  3. 范围条件之后的列，通常不能继续用于索引过滤。
  4. 如果查询需要排序，尽量让索引顺序同时满足 ORDER BY。
  5. 高频查询可以考虑覆盖索引，减少回表。

例如：

SELECT id, order_no, amount
FROM orders
WHERE user_id = ?
  AND status = ?
  AND create_time >= ?
ORDER BY create_time DESC
LIMIT 20;

可能适合的索引是：

CREATE INDEX idx_user_status_time
ON orders (user_id, status, create_time);

如果查询只需要 id、order_no、amount，还可以评估是否值得扩展成覆盖索引：

CREATE INDEX idx_user_status_time_cover
ON orders (user_id, status, create_time, order_no, amount);

覆盖索引能减少回表，但会让索引更大，写入更慢。是否值得，需要看查询频率和写入压力。

4.3 常见索引失效场景

常见问题包括：

  • 对索引列使用函数：DATE(create_time) = '2026-07-02'
  • 隐式类型转换：字符串列用数字条件查询。
  • 前导模糊匹配：LIKE '%abc'
  • 范围条件之后继续期待后续列生效。
  • OR 条件两侧索引设计不一致。
  • 统计信息不准导致优化器选错索引。

很多“索引失效”不是 MySQL 不使用索引，而是索引用得不充分。分析时要看扫描行数，而不是只看 key 字段有没有值。

5. InnoDB 存储与内存结构

SQL 优化之所以不能停留在语法层面，是因为执行成本最终会落到页读取、缓存命中、索引维护和磁盘刷写等存储引擎行为上。InnoDB 的若干核心结构决定了许多优化结论的边界。

5.1 页与行记录

InnoDB 以页为基本管理单位，默认页大小为 16KB。数据页中包含页头、用户记录、空闲空间、页目录以及校验信息。用户记录按主键顺序组织，页目录用于支持页内查找，叶子页之间通过指针形成有序链表。

这一结构意味着：查询并不是逐行从磁盘读取，而是以页为单位读取。若查询条件无法有效缩小范围，即使最终只返回少量记录，也可能读取大量无关页。

5.2 Buffer Pool

Buffer Pool 是 InnoDB 最核心的内存结构。数据页和索引页在访问时会先进入 Buffer Pool；命中时直接从内存返回，未命中时才访问磁盘。

相关指标如下：

SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_read_requests';
SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_reads';

粗略命中率可以表示为：

命中率 = 1 - Innodb_buffer_pool_reads / Innodb_buffer_pool_read_requests

Buffer Pool 命中率下降并不必然意味着只能扩容内存。低效索引、全表扫描、深分页、大量冷数据混在主表中，都会降低缓存的有效性。因此，缓存治理与 SQL 改写、索引设计、冷热分离之间存在直接关联。

5.3 Change Buffer

对于非唯一二级索引的写入，如果目标索引页不在 Buffer Pool 中，InnoDB 可以先将变更记录到 Change Buffer，待后续读取该页或后台合并时再写入真实索引页。

该机制通过延迟合并减少随机读，但只适用于非唯一二级索引。唯一索引必须立即判断唯一性，因此无法以相同方式推迟校验。

由此可得出一个工程推论：索引不仅影响查询，也影响写入路径。写多读少的表若存在大量二级索引或唯一索引，写入成本会显著上升。

5.4 自适应哈希索引

Adaptive Hash Index 是 InnoDB 针对部分高频等值访问自动建立的内存哈希结构。它可以在特定访问模式下减少 B+Tree 查找成本，但也会占用内存并引入并发维护开销。

因此，自适应哈希索引不是必然收益项。是否开启应依据实际负载、等待事件和性能指标判断，而不应作为固定调优动作。

6. 大表治理与架构级优化

当单表规模持续增长时，分库分表并不应作为首选方案。其原因在于，分库分表会引入路由规则、分布式事务、跨分片查询、全局唯一 ID、扩容迁移和数据倾斜等额外复杂度。若 SQL、索引、冷热数据和访问模式仍存在明显优化空间，过早分片反而会放大系统复杂度。

较为稳妥的治理顺序如下：

SQL 和索引优化
    ↓
字段裁剪和垂直拆分
    ↓
冷热数据归档
    ↓
分区表或历史表
    ↓
读写分离
    ↓
分库分表

6.1 冷热分离

大量业务表的数据访问具有明显时间局部性。例如订单表的大部分在线查询可能集中于最近数月。若主表长期承载多年历史数据，索引体积、缓存污染和归档成本都会持续增加。

一种可行的数据生命周期分层方式如下：

  • 主表保留热数据。
  • 历史表或归档库存放温数据。
  • 离线系统或对象存储保存冷数据。
  • 查询入口根据时间范围路由到不同数据源。

归档任务应具备分批、限速、可暂停和可校验能力，并持续监控主从延迟、实例负载和业务错误率。

6.2 深分页

深分页是慢 SQL 的高发来源：

SELECT *
FROM orders
ORDER BY id
LIMIT 1000000, 20;

数据库需要扫描并跳过前 1000000 行，再返回 20 行。更合理的方式是基于游标或业务有序键进行翻页：

SELECT *
FROM orders
WHERE id > ?
ORDER BY id
LIMIT 20;

若业务必须支持任意跳页，应评估搜索引擎、页码边界缓存或交互方式调整，而不宜将全部成本转嫁给在线 MySQL 实例。

6.3 在线 DDL

大表加索引、改字段、改表结构都有风险。即使 MySQL 支持 Online DDL，也可能因为 MDL 锁等待、IO 压力、主从延迟而影响业务。

生产环境中常用 gh-ost 或 pt-online-schema-change 降低大表 DDL 风险。此类工具通过影子表、增量同步和最终切换减少锁表时间，但仍需满足以下约束：

  • 低峰执行。
  • 限制批次大小。
  • 监控主从延迟。
  • 准备回滚方案。
  • 先在预发或影子环境验证。

7. WAL、Redo Log 与 Undo Log

写入性能与事务持久性之间的平衡，主要由 InnoDB 日志系统承担。InnoDB 能够在保证事务持久性的同时维持较高写入吞吐，核心原因在于 WAL 机制。

7.1 WAL：先写日志，再写数据

WAL 即 Write-Ahead Logging，含义是“先写日志，再写数据”。事务提交时，InnoDB 不需要立即将数据页随机写回磁盘，而是先将物理修改写入 redo log：

事务写入流程
1. 修改 Buffer Pool 中的数据页，数据页变成脏页
2. 将物理修改写入 Redo Log Buffer
3. 事务提交时，将 Redo Log 按策略刷到磁盘
4. 返回客户端提交成功
5. 后台线程异步把脏页刷回数据文件

该设计将大量随机写转化为顺序写。只要 redo log 已经持久化，即使脏页尚未刷回数据文件，MySQL 崩溃后也可以通过 redo log 重放完成恢复。

7.2 Redo Log 环形空间

Redo Log 可以抽象为循环使用的日志空间：

checkpoint ---------------- write pos
     ↑                         ↑
已刷回数据文件的位置        当前写入位置
  • write pos：当前写入位置。
  • checkpoint：已经刷回数据文件的位置。

当写入速度持续高于脏页刷新速度时，write pos 会逐渐逼近 checkpoint。一旦 redo log 可用空间不足，前台写入会等待刷脏页推进 checkpoint。因此，线上写入抖动不应仅观察 SQL 本身，还应同时检查 redo log 空间、脏页比例和磁盘刷写能力。

7.3 innodb_flush_log_at_trx_commit

这个参数影响事务提交时 redo log 的刷盘策略：

值 行为 性能 数据安全
1 每次事务提交都尽量 fsync redo log 最低 最安全
2 每次提交写入 OS 缓存，每秒 fsync 较高 OS 崩溃可能丢约 1 秒
0 每秒写入并 fsync，提交时不主动写 最高 MySQL 崩溃也可能丢约 1 秒

核心交易链路通常应保持 1。日志、统计或可重建数据是否允许放宽，应由业务可接受的数据丢失窗口决定。

7.4 Undo Log 与版本链

Undo Log 记录数据修改前的旧版本，主要有两个作用：

  • 事务回滚时恢复旧值。
  • MVCC 快照读时读取历史版本。

一行数据被多次修改后，会形成版本链：

当前行: trx_id=300, roll_pointer
   ↓
Undo V2: trx_id=200
   ↓
Undo V1: trx_id=100

当某个事务读取当前版本不可见时，InnoDB 会沿 undo 版本链向前查找，直到定位到对该事务可见的历史版本。

长事务会阻止旧版本清理，导致 undo 表空间膨胀、版本链延长，并进一步增加快照读成本。因此，慢 SQL 排查除执行计划外，还应检查是否存在长期未提交事务。

8. 事务隔离、MVCC 与锁

数据库系统必须在并发读写条件下同时满足正确性与吞吐量要求。事务隔离、MVCC 与锁机制正是围绕这一矛盾建立的并发控制体系。

8.1 隔离级别

隔离级别 脏读 不可重复读 幻读 说明
Read Uncommitted 可能 可能 可能 很少用于业务系统
Read Committed 不会 可能 可能 并发度较好
Repeatable Read 不会 通常不会 InnoDB 通过 MVCC + 锁机制处理 MySQL 常见默认值
Serializable 不会 不会 不会 并发度最低

Repeatable Read 提供更稳定的事务快照，但在范围更新场景下可能引入更大的锁范围。Read Committed 的并发度通常更高，但需要应用层通过唯一约束、乐观锁、状态机和幂等机制补足业务一致性。

8.2 ReadView

MVCC 的核心结构是 ReadView。执行快照读时，InnoDB 会生成读视图，其中包含当前活跃事务列表等信息，并据此判断记录版本的可见性。

简化可见性判断：

如果版本事务 ID 是当前事务自己：可见
如果版本事务 ID 小于最小活跃事务 ID：可见
如果版本事务 ID 大于等于下一个待分配事务 ID：不可见
如果版本事务 ID 仍在活跃事务列表中：不可见
否则：可见

RC 和 RR 的核心差异在 ReadView 生成时机：

隔离级别 ReadView 生成时机 效果
RC 每次 SELECT 都生成新的 ReadView 每次读都能看到最新已提交数据
RR 第一次快照读生成，事务内复用 同一事务内快照稳定

8.3 快照读与当前读

普通 SELECT 是快照读，不读取最新版本，而是根据 ReadView 读取可见版本。

以下操作属于当前读：

SELECT ... FOR UPDATE;
SELECT ... LOCK IN SHARE MODE;
UPDATE ...;
DELETE ...;
INSERT ...;

当前读读取最新版本，并可能加锁。很多锁等待问题都发生在当前读上，尤其是范围更新、缺少索引的更新和删除。

8.4 行锁、间隙锁与 Next-Key Lock

InnoDB 的锁加在索引上。常见锁包括：

  • Record Lock：锁住索引记录。
  • Gap Lock：锁住索引记录之间的间隙。
  • Next-Key Lock：Record Lock + Gap Lock。

由此可以得到以下结论：

  • 没有合适索引时，更新可能扫描大量记录，锁范围被放大。
  • 范围条件在 RR 下可能产生间隙锁。
  • 唯一索引等值命中时，锁范围通常更小。

排查锁等待可以看：

SELECT *
FROM sys.innodb_lock_waits\G

9. 查询执行引擎与执行计划

执行计划是连接 SQL 文本与存储引擎执行行为之间的关键证据。它并不等同于最终运行结果，但能够揭示优化器选择的访问路径、连接顺序、扫描规模和额外操作。

9.1 SQL 执行流程

客户端发送 SQL
  ↓
连接器处理连接和权限
  ↓
解析器做词法、语法分析
  ↓
预处理器检查表、字段、权限
  ↓
优化器选择执行计划
  ↓
执行器调用 InnoDB 接口
  ↓
InnoDB 读取索引页和数据页
  ↓
返回结果

优化器依赖统计信息估算成本。统计信息不准、字段分布倾斜、参数差异大时，优化器可能选错索引。此时可以考虑 ANALYZE TABLE 更新统计信息，但生产上要评估影响。

9.2 EXPLAIN 重点字段

字段 关注点
type 访问类型，判断是全表扫描、范围扫描还是等值索引
key 实际使用的索引
rows 预估扫描行数
filtered 过滤后剩余比例
Extra 是否排序、临时表、覆盖索引、索引条件下推

常见访问类型：

type 含义
ALL 全表扫描
index 全索引扫描
range 范围扫描
ref 非唯一索引等值查询
eq_ref JOIN 中通过唯一索引匹配一行
const 主键或唯一索引常量查询

常见 Extra：

Extra 含义
Using index 覆盖索引，不需要回表
Using where 存储引擎返回后仍需过滤
Using index condition 索引条件下推
Using filesort 需要额外排序
Using temporary 使用临时表

9.3 EXPLAIN ANALYZE

MySQL 8.0 支持：

EXPLAIN ANALYZE SELECT ...

它会实际执行 SQL，并输出每个执行节点的真实耗时和行数。它比普通 EXPLAIN 更接近真实情况，但也因为会执行 SQL，生产环境要谨慎使用。

10. SQL 优化实战

10.1 避免 SELECT *

SELECT * 的问题：

  • 读取不必要字段，增加 IO 和网络传输。
  • 大字段可能触发额外页读取。
  • 无法使用覆盖索引。
  • 表结构变化可能影响应用。

列表页、接口响应、导出任务都应该明确字段。

10.2 深分页

典型慢查询：

SELECT *
FROM orders
ORDER BY id
LIMIT 1000000, 20;

数据库需要跳过前 1000000 行，再返回 20 行。更好的方式是游标分页：

SELECT *
FROM orders
WHERE id > ?
ORDER BY id
LIMIT 20;

如果业务必须支持跳页，可以考虑缓存页码边界、搜索引擎或调整交互方式。

10.3 JOIN 优化

JOIN 优化重点：

  • 小结果集驱动大表。
  • 被驱动表的 JOIN 字段要有索引。
  • 尽量先过滤再 JOIN。
  • 控制返回字段，避免宽行 JOIN。

示例：

SELECT o.id, o.amount, u.name
FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.status = ?
  AND o.create_time >= ?;

这里 orders 应该能通过合适索引先缩小范围，users.id 是主键，JOIN 成本才可控。

10.4 排序与分组

ORDER BY 和 GROUP BY 如果不能利用索引，可能产生 filesort 或临时表。

优化方向：

  • 让索引顺序匹配过滤条件和排序字段。
  • 先过滤再排序。
  • 高频统计用汇总表或离线预聚合。
  • 大报表不要直接扫在线业务明细表。

11. 线上排查路径

11.1 个别 SQL 慢

排查顺序：

  1. 从阿里云慢 SQL 监控拿 SQL、耗时、扫描行数、执行次数。
  2. 查表结构、索引、数据量。
  3. 使用 EXPLAIN 分析执行计划。
  4. 判断是否全表扫描、排序、临时表、索引选择错误。
  5. 检查是否有锁等待。
  6. 改 SQL 或索引。
  7. 验证执行计划和耗时。

11.2 所有 SQL 都慢

如果所有 SQL 都变慢，往往不是单条 SQL 的问题：

  • CPU 是否被打满。
  • 磁盘 IO 是否打满。
  • Buffer Pool 命中率是否下降。
  • 连接数是否打满。
  • 是否有长事务。
  • 是否有 DDL 等待 MDL 锁。
  • 主从延迟是否异常。

一种典型传导链路是：新上线 SQL 触发全表扫描，磁盘 IO 被快速耗尽，Buffer Pool 热数据被污染，随后正常 SQL 的响应时间也被整体拉高。

11.3 CPU 飙高

先看当前活跃 SQL：

SELECT *
FROM information_schema.PROCESSLIST
WHERE COMMAND != 'Sleep'
ORDER BY TIME DESC
LIMIT 20;

如果某类 SQL 大量堆积且确认有问题，可以先在业务层限流、回滚发布或临时 kill 相关线程。根因通常包括缺索引、复杂排序、复杂 JOIN、函数计算、锁竞争。

12. 专题后续展开方向

本文作为数据库专题的总论，后续可以沿以下方向展开为独立文章：

  1. 慢 SQL 分析实战：慢查询日志、EXPLAIN、EXPLAIN ANALYZE、performance_schema。
  2. 索引设计方法论：联合索引、覆盖索引、索引下推、最左前缀、冗余索引。
  3. 执行计划详解：系统解释 type、key、rows、filtered、Extra 等字段的工程含义。
  4. 事务与锁：MVCC、ReadView、Next-Key Lock、死锁、长事务。
  5. InnoDB 存储机制：页、行格式、Buffer Pool、Change Buffer、AHI。
  6. 日志系统：Redo Log、Undo Log、Binlog、两阶段提交、崩溃恢复。
  7. 大表治理：冷热分离、归档、分区表、在线 DDL。
  8. 数据库架构演进：读写分离、分库分表、TiDB、ES、OLAP。

这样组织后，数据库专题会从“零散知识点”变成一条清晰路线：先能分析慢 SQL，再理解索引和执行计划，最后补上 InnoDB 内核和架构治理。

13. 常用诊断 SQL

用于观察当前正在执行的 SQL：

SELECT *
FROM information_schema.PROCESSLIST
WHERE COMMAND != 'Sleep'
ORDER BY TIME DESC
LIMIT 20;

用于分析锁等待关系：

SELECT *
FROM sys.innodb_lock_waits\G

用于识别长事务：

SELECT trx_id,
       trx_state,
       trx_started,
       TIMESTAMPDIFF(SECOND, trx_started, NOW()) AS duration_sec,
       trx_query
FROM information_schema.INNODB_TRX
ORDER BY trx_started ASC;

用于检查冗余索引：

SELECT *
FROM sys.schema_redundant_indexes;

用于检查未使用索引：

SELECT *
FROM sys.schema_unused_indexes;

用于评估表与索引空间占用：

SELECT table_name,
       ROUND(data_length / 1024 / 1024, 2) AS data_mb,
       ROUND(index_length / 1024 / 1024, 2) AS index_mb,
       table_rows
FROM information_schema.TABLES
WHERE table_schema = 'your_db'
ORDER BY data_length DESC;

14. 结论

数据库优化并非若干孤立技巧的简单集合，而是一套证据驱动的闭环能力：通过监控发现问题，通过执行计划定位瓶颈，通过存储引擎机制解释成本，通过可控变更验证效果，并通过持续治理降低问题复发概率。

对于应用开发者而言，慢 SQL 分析、执行计划阅读和索引设计应当构成数据库能力的第一层基础。在此基础上继续深入 Buffer Pool、Redo / Undo Log、MVCC、锁机制、大表治理和数据库架构演进，才能形成相对完整的 MySQL 性能优化体系。
