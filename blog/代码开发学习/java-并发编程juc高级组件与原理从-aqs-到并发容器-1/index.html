Java 并发编程（JUC）高级组件与原理：从 AQS 到并发容器

摘要

本文档旨在深入剖析 Java 并发工具包（JUC, Java Util Concurrent）中的核心高级组件与原理。我们将秉持**“合理符合实际、准确细节完备”**的原则，从并发编程的基石 JVM 优化出发，详细解析 AQS（AbstractQueuedSynchronizer）的底层机制，并扩展到 ReentrantLock、并发容器（如 ConcurrentHashMap）和 线程池 的实现细节，力求帮助读者彻底理解 JUC 的设计哲学和高性能秘密。

1. 并发基础：JVM 层的优化与保障

1.1 synchronized 的锁升级机制

synchronized 关键字是 Java 中最基本的同步机制，但在 JDK 1.6 之后，其性能得到了大幅优化，不再是传统意义上的“重量级锁”。JVM 引入了锁升级机制，以适应不同的竞争情况，从而减少锁竞争带来的性能开销。

锁的级别从低到高依次为：无锁 $\to$ 偏向锁 $\to$ 轻量级锁 $\to$ 重量级锁。

锁状态 适用场景 实现原理 性能特点
偏向锁 (Biased Locking) 几乎没有竞争，锁总是由同一个线程获得。 锁对象头中记录持有线程ID。线程再次进入同步块时，只需比较线程ID，无需 CAS 操作。 性能最高。适用于单线程反复进入同步块。
轻量级锁 (Lightweight Locking) 少量竞争，线程交替进入同步块。 线程在栈帧中创建锁记录（Lock Record），通过 CAS 尝试将对象头中的 Mark Word 替换为指向锁记录的指针。 性能较高。适用于线程交替执行，没有阻塞。
重量级锁 (Heavyweight Locking) 激烈竞争，线程需要阻塞等待。 锁膨胀为重量级锁，指向对象监视器（Monitor）。未获取锁的线程进入阻塞状态，涉及操作系统内核态切换。 性能最低。但能保证线程安全和公平性。

JVM 优化： JVM 还会进行锁消除（JIT 编译器发现不可能存在竞争的锁，直接消除）和锁粗化（将连续的加锁、解锁操作合并为一个更大的同步块），进一步提升 synchronized 的性能。

1.2 volatile 与内存屏障

volatile 关键字是 JUC 中许多组件（如 AQS 的 state 变量）实现可见性的底层保障。它主要解决两个问题：可见性和有序性。

  • 可见性：当一个线程修改了 volatile 变量的值，新值会立即同步到主内存，并且其他线程读取该变量时，会强制从主内存刷新。
  • 有序性：volatile 通过插入内存屏障（Memory Barrier）来禁止指令重排序。
内存屏障类型 插入位置 作用
StoreStore 屏障 在 volatile 写操作之前。 确保 volatile 写之前的操作，不会被重排序到 volatile 写之后。
StoreLoad 屏障 在 volatile 写操作之后。 确保 volatile 写操作对其他线程可见后，才执行后续的读操作。
LoadLoad 屏障 在 volatile 读操作之后。 确保 volatile 读操作之前和之后的读操作，不会被重排序。
LoadStore 屏障 在 volatile 读操作之后。 确保 volatile 读操作之前和之后的写操作，不会被重排序。

核心价值： volatile 保证了变量的修改能及时被其他线程看到，是实现轻量级同步和无锁编程的基础。

2. 并发基石：AQS（AbstractQueuedSynchronizer）

AQS 是 JUC 中许多同步器（如 ReentrantLock、Semaphore、CountDownLatch 等）的抽象基类。它提供了一个FIFO（先进先出）的等待队列和一套原子操作，用于管理线程的阻塞和唤醒。

2.1 AQS 的核心组成

AQS 的设计围绕三个核心要素：

核心要素 作用 底层实现
State 状态 用于表示同步状态，如锁的持有次数、信号量的可用资源数等。 volatile int state，通过 CAS（Compare-And-Swap）原子操作修改。
CLH 队列 用于存储等待获取同步状态的线程。 双向链表，每个节点（Node）封装了一个等待的线程。
CAS 机制 用于安全地修改 state 变量和操作 CLH 队列。 Unsafe 类提供的原子操作，确保多线程环境下的可见性和原子性。

2.2 CLH 队列与线程阻塞/唤醒

CLH 队列（Craig, Landin, and Hagersten lock）是 AQS 实现线程等待和唤醒的关键。

  • 入队（自旋与阻塞）：当线程尝试获取同步状态失败时，它会被封装成一个 Node 节点，通过 CAS 尾插法加入到 CLH 队列的尾部。
    • 自旋：新入队的线程不会立即阻塞，而是会自旋检查其前驱节点的状态。
    • 阻塞：只有当前驱节点是 SIGNAL 状态时，线程才会调用 LockSupport.park() 方法进入等待状态（阻塞），释放 CPU 资源。
  • 出队（唤醒）：当持有锁的线程释放同步状态时，它会唤醒 CLH 队列中的头节点的后继节点。
    • 唤醒：被唤醒的线程会调用 LockSupport.unpark() 方法，从等待状态中恢复，重新尝试获取同步状态。

核心细节： AQS 巧妙地利用了 “自旋 + park/unpark” 机制，避免了线程在内核态和用户态之间频繁切换的开销，只有在确定需要等待时才阻塞，提高了效率。

2.3 ReentrantLock：公平与非公平的抉择

ReentrantLock 是 AQS 的典型应用，它提供了两种锁模式：

模式 特性 性能考量
非公平锁（默认） 线程尝试获取锁时，不会检查等待队列中是否有线程等待。如果锁是空闲的，新线程可以直接抢占。 性能更高。减少了线程切换和入队出队的开销，但可能导致队列中的线程饥饿。
公平锁 线程尝试获取锁时，会检查等待队列。如果队列中有线程等待，新线程必须排队。 性能略低。保证了线程获取锁的顺序性，避免饥饿，但增加了额外的检查开销。

实践建议： 在绝大多数高并发场景中，应优先使用非公平锁，因为其吞吐量更高。只有在对线程公平性有严格要求的场景下，才考虑使用公平锁。

3. 并发容器：ConcurrentHashMap 与 CopyOnWrite

3.1 ConcurrentHashMap（JDK 1.8）

ConcurrentHashMap 是 Java 中使用最广泛的并发 Map，其设计经历了从分段锁（JDK 1.7）到 CAS + Synchronized（JDK 1.8）的演变。

3.1.1 JDK 1.8 的核心结构

JDK 1.8 的 ConcurrentHashMap 采用数组 + 链表 + 红黑树的结构，其并发控制机制是：

  • 写操作（put/remove）：只对当前操作的桶（Bucket）加锁。锁的粒度是桶级别的，使用的是 synchronized 关键字。
  • 读操作（get）：完全无锁。通过 volatile 关键字保证可见性，直接读取数据。
  • 扩容：采用多线程协助扩容机制，将扩容过程分散到多个线程的写操作中，避免了单线程扩容带来的长时间阻塞。

3.1.2 锁的细节：CAS 与 Synchronized

  1. 初始化/扩容：使用 CAS 机制来控制对数组的初始化和扩容操作，确保只有一个线程能成功执行。
  2. 链表/红黑树操作：使用 synchronized 锁住当前 Node 或 TreeBin 的头节点，实现对桶内数据的并发控制。

设计哲学： 读操作完全无锁，写操作只锁住必要的最小范围，最大化了并发性能。

3.2 CopyOnWriteArrayList

CopyOnWriteArrayList（写时复制列表）是一种特殊的并发容器，适用于读多写少的场景。

  • 写操作（add/set/remove）：
    1. 获取一个全局锁（ReentrantLock）。
    2. 复制底层数组。
    3. 在新数组上执行修改操作。
    4. 将底层数组的引用指向新数组。
    5. 释放锁。
  • 读操作（get）：完全无锁。直接读取当前数组的引用。

数据一致性： 读操作读取的是旧数组，因此 CopyOnWriteArrayList 提供了最终一致性。它牺牲了写操作的性能（因为涉及数组复制），换取了读操作的极致性能。

4. 原子操作与 CAS 机制

4.1 CAS（Compare-And-Swap）

CAS 是一种乐观锁机制，由 CPU 硬件指令支持，用于实现无锁（Lock-Free）编程。

  • 操作过程：它包含三个操作数：
    1. V（Value）：内存位置的当前值。
    2. A（Expected）：预期的旧值。
    3. B（New）：要写入的新值。
  • 逻辑：只有当内存位置 V 的值等于预期值 A 时，才将 V 的值更新为 B。否则，不做任何操作。

优势： 避免了使用重量级的锁（如 synchronized），减少了线程上下文切换的开销。

4.2 ABA 问题与解决方案

CAS 机制存在一个著名的漏洞：ABA 问题。

  • 问题描述：
    1. 线程 T1 读取 V 的值为 A。
    2. 线程 T2 将 V 的值修改为 B，然后又改回 A。
    3. 线程 T1 执行 CAS 操作时，发现 V 的值仍然是 A，认为没有发生变化，成功执行更新。
  • 危害：虽然值没有变，但过程中的变化可能导致逻辑错误（例如，链表中的节点被删除又重新添加）。

解决方案： 使用 AtomicStampedReference。

  • 原理：在 CAS 比较时，不仅比较值 V，还引入一个版本号（Stamp）。
  • 逻辑：只有当 V 的值等于预期值 A 且版本号等于预期版本号时，才执行更新，并将版本号加 1。这样，即使值从 A 变为 B 又变回 A，版本号也已经改变，CAS 操作会失败，从而避免了 ABA 问题。

5. 线程池核心原理：ThreadPoolExecutor

ThreadPoolExecutor 是 JUC 中管理线程的核心组件，其设计目标是复用线程，避免线程频繁创建和销毁的开销。

5.1 线程池的七大参数与工作流程

线程池的工作流程围绕以下七个核心参数展开：

参数 作用
corePoolSize 核心线程数。线程池中始终保持存活的线程数量。
maximumPoolSize 最大线程数。线程池允许创建的最大线程数量。
keepAliveTime 非核心线程的空闲存活时间。
unit keepAliveTime 的时间单位。
workQueue 任务队列。用于存放等待执行的任务。
threadFactory 线程工厂。用于创建新线程。
handler 拒绝策略。当任务队列和线程池都满时，采取的拒绝措施。

工作流程（优先级）：

  1. 核心线程：当新任务提交时，如果当前运行的线程数小于 corePoolSize，则创建核心线程执行任务。
  2. 任务队列：如果当前运行的线程数等于 corePoolSize，新任务会被放入 workQueue 等待。
  3. 非核心线程：如果任务队列已满，且当前运行的线程数小于 maximumPoolSize，则创建非核心线程执行任务。
  4. 拒绝策略：如果线程数达到 maximumPoolSize 且任务队列已满，则执行拒绝策略。

5.2 任务队列的选择与影响

workQueue 的选择直接影响线程池的行为和性能：

队列类型 特性 影响
LinkedBlockingQueue 无界队列（默认容量为 Integer.MAX_VALUE）。 任务会无限排队，导致线程数永远不会超过 corePoolSize。maximumPoolSize 参数失效。适用于任务提交速度稳定且可控的场景。
ArrayBlockingQueue 有界队列。 任务排队达到上限后，会触发创建非核心线程，直到达到 maximumPoolSize。适用于需要控制资源消耗的场景。
SynchronousQueue 不存储元素的阻塞队列。 任务提交后，必须等待一个工作线程立即接收。如果当前没有空闲线程，则创建新线程（直到 maximumPoolSize），否则执行拒绝策略。适用于任务提交速度极快，且需要快速响应的场景。

5.3 拒绝策略（Handler）

当线程池和任务队列都满时，ThreadPoolExecutor 会执行拒绝策略。JUC 提供了四种内置策略：

  1. AbortPolicy（默认）：直接抛出 RejectedExecutionException 异常。
  2. CallerRunsPolicy：由提交任务的线程（Caller）自己执行该任务。
  3. DiscardPolicy：直接丢弃任务，不抛出异常。
  4. DiscardOldestPolicy：丢弃任务队列中最旧的任务，然后尝试重新提交当前任务。

实践建议： 在生产环境中，应根据业务需求自定义拒绝策略，例如将任务记录到日志或持久化到数据库，以确保任务不会无声无息地丢失。

6. 总结

JUC 的设计哲学是将并发控制的复杂性封装在底层，通过 AQS、CAS 等机制，为上层应用提供了高效、可靠的并发工具。理解这些底层原理，不仅能帮助我们正确使用 JUC 组件，更能指导我们在遇到并发问题时，快速定位和解决问题。

知识体系回顾：

  • JVM 基础：synchronized 锁升级、volatile 内存屏障。
  • 并发基石：AQS 队列、状态、park/unpark。
  • 核心工具：ReentrantLock、ConcurrentHashMap、ThreadPoolExecutor。
  • 原子操作：CAS、ABA 问题与 AtomicStampedReference。
