生产问题复盘-布隆过滤器误判导致连接池耗尽

问题概述

问题类型：数据库连接池耗尽

影响范围：用户注册功能

严重程度：高

本文重点：展示从现象到根因的完整排查思路和过程

一、问题现象

1.1 用户反馈

  • 用户注册失败：部分用户反馈注册时提示”系统繁忙，请稍后重试”
  • 响应超时：注册接口响应时间过长，甚至超时
  • 服务异常：部分功能无法正常使用

1.2 系统监控告警

  • 数据库连接池告警：连接池使用率达到100%，无法获取新连接
  • 接口超时告警：用户注册接口超时率上升
  • 线程数告警：应用线程数异常增长

1.3 初步判断

看到的现象：

  • 数据库连接池被占满
  • 用户注册功能异常
  • 服务响应变慢

初步猜测（此时还不知道真正原因）：

  • 可能是数据库连接泄漏？
  • 可能是某个慢查询？
  • 可能是事务未提交？
  • 可能是死锁？

二、排查思路

2.1 排查方法论

排查原则：

  1. 从现象出发：先看监控数据，找到异常指标
  2. 层层深入：从表面现象找到深层原因
  3. 验证假设：每个假设都要有证据支撑
  4. 定位根因：找到问题的根本原因，而不是表面症状

2.2 排查路径设计

现象：数据库连接池耗尽
    ↓
问题1：为什么连接池会被占满？
    ↓
假设1：连接泄漏（连接未释放）
    ↓
验证1：查看连接池监控，连接是否正常释放？
    ↓
问题2：为什么连接未释放？
    ↓
假设2：事务执行时间过长
    ↓
验证2：查看事务监控，是否有长时间运行的事务？
    ↓

    ↓

    ↓
定位：找到耗时操作
    ↓

@Transactional
public void registerUser(UserRegisterDTO dto) {

3.5 第五步：分析循环逻辑

操作：分析while循环的执行逻辑

代码逻辑分析：

do {
    clientCode = generateClientCode();  // 生成新的clientCode
} while (bloomFilter.mightContain(clientCode));  // 如果布隆过滤器判断"可能存在"，则继续循环

问题分析：

  1. 循环条件：
    • bloomFilter.mightContain(clientCode) 返回 true 时，继续循环
    • 返回 false 时，跳出循环
  2. 正常情况：
    • 生成的clientCode实际上不存在
    • 布隆过滤器判断”不存在”（返回false）
    • 循环1次就退出
  3. 异常情况：
    • 生成的clientCode实际上不存在
    • 但布隆过滤器误判为”可能存在”（返回true）
    • 导致继续循环，重新生成
    • 如果误判率高，循环次数会大大增加

关键问题：

  • 为什么布隆过滤器会误判？
  • 误判率有多高？

下一步：需要分析布隆过滤器的配置和使用情况

关键发现：

  • 实际使用量是初始容量的5倍
  • 问题：容量不足会导致什么后果？

下一步：需要理解布隆过滤器的误判率机制

3.7 第七步：理解误判率机制

操作：查阅布隆过滤器原理，分析误判率

布隆过滤器误判率公式：

误判率 ≈ (1 - e^(-kn/m))^k

其中：
- m：位数组长度（容量）
- k：哈希函数个数
- n：已插入元素个数

计算分析：

  1. 设计时的误判率（容量1万，插入1万）：

    误判率 ≈ 1%（设计值）

  2. 实际误判率（容量1万，插入5万）：

    当 n（已插入元素）接近或超过 m（容量）时：
- 位数组几乎全为1
- 误判率急剧上升
- 可能达到 10%+ 甚至更高

验证：

  • 如果误判率为10%，那么：
  • 平均循环次数 = 1 / (1 - 0.1) ≈ 1.11次
  • 如果误判率为20%，平均循环次数 = 1 / (1 - 0.2) = 1.25次
  • 极端情况下，可能循环数十次甚至上百次

关键发现：

  • 容量不足导致误判率急剧上升
  • 误判率高导致循环次数增加
  • 循环次数增加导致Redis查询频率异常高（3亿次/秒）

下一步：需要分析为什么会导致连接池耗尽

3.8 第八步：分析连接池耗尽的原因

操作：分析事务超时和连接泄漏

代码分析：

@Transactional  // 事务注解
public void registerUser(UserRegisterDTO dto) {
    // while循环可能执行很长时间
    do {
        clientCode = generateClientCode();
    } while (bloomFilter.mightContain(clientCode));  // 每次循环都查询Redis

    // 如果循环时间过长，事务可能超时
    // 但线程还在执行，数据库连接未释放
    userService.save(user);
}

问题链条分析：

1. 用户发起注册请求
   ↓
2. 进入事务方法，获取数据库连接
   ↓
3. while循环执行（由于误判率高，循环次数多，如10+次）
   ↓
4. 每次循环都查询Redis（布隆过滤器）
   ↓
5. 循环时间过长（如5秒+），超过事务超时时间（如3秒）
   ↓
6. 事务超时，客户端连接断开
   ↓
7. 但线程还在执行while循环，数据库连接未释放
   ↓
8. 多个请求同时发生，多个连接被占用
   ↓
9. 数据库连接池被占满

验证：

  • 查看事务超时配置：3秒
  • 查看接口响应时间：30秒+（远超事务超时时间）
  • 查看线程堆栈：大量线程处于RUNNABLE状态，正在执行while循环
  • 查看连接状态：大量连接长时间占用，未释放

关键发现：

  • 事务超时后，线程仍在执行，连接未释放
  • 多个并发请求导致多个连接被占用
  • 连接池容量有限，很快被占满

根因确认：

  • 根本原因：布隆过滤器容量不足 → 误判率高 → 循环次数增加 → 事务超时 → 连接泄漏 → 连接池耗尽

四、问题验证（如何确认？）

4.1 验证误判率假设

方法：查看布隆过滤器的实际使用情况

检查结果：
- 布隆过滤器初始容量：1万
- 当前已插入元素数量：5万
- 容量使用率：500%（严重超载）

结论：

  • 容量确实严重不足
  • 误判率确实会急剧上升

4.2 验证循环次数假设

方法：分析Redis查询频率

监控数据：
- Redis Key调用频率：3亿次/秒
- 正常注册量：假设1000次/秒
- 平均循环次数 = 3亿 / 1000 = 30万次（异常高！）

实际分析：
- 如果每个注册循环10次，那么：
  - 3亿次/秒 ÷ 10次 = 3000万次注册/秒（不可能）
- 实际情况：
  - 大量请求在循环中，导致查询频率异常高
  - 循环次数确实异常高

结论：

  • 循环次数确实异常高
  • Redis查询频率异常高

4.3 验证连接泄漏假设

方法：查看连接池和线程状态

检查结果：
- 连接池使用率：100%
- 等待获取连接的线程数：50+
- 线程堆栈：大量线程处于RUNNABLE状态，正在执行while循环
- 连接状态：大量连接长时间占用（300秒+），未释放

结论：

  • 连接确实未释放
  • 线程确实在执行长时间操作

五、根因分析（为什么会出现？）

5.1 问题链条

业务增长（客户数量从1万增长到5万）
    ↓
布隆过滤器容量不足（初始容量1万，实际使用5万）
    ↓
误判率急剧上升（从1%上升到10%+）
    ↓
while循环次数增加（从1次增加到10+次）
    ↓
事务执行时间过长（从100ms增加到5秒+）
    ↓
事务超时（超过3秒超时时间）
    ↓
客户端连接断开，但线程仍在执行
    ↓
数据库连接未释放
    ↓
多个并发请求导致多个连接被占用
    ↓
数据库连接池被占满
    ↓
服务不可用

5.2 根本原因

直接原因：

  • 布隆过滤器容量不足，导致误判率高

深层原因：

  1. 容量规划不足：没有考虑业务增长，初始容量设置过小
  2. 监控缺失：没有监控布隆过滤器的使用率
  3. 代码设计缺陷：在事务中执行可能耗时的循环操作
  4. 缺乏保护机制：循环没有最大次数限制

六、解决方案

6.1 立即修复

修改布隆过滤器容量：

// 修改前
BloomFilter<String> bloomFilter = BloomFilter.create(
    Funnels.stringFunnel(Charset.defaultCharset()),
    10000,  // 初始容量：1万
    0.01    // 误判率：1%
);

// 修改后
BloomFilter<String> bloomFilter = BloomFilter.create(
    Funnels.stringFunnel(Charset.defaultCharset()),
    100000, // 初始容量：10万（预留2倍空间）
    0.01    // 误判率：1%
);

容量设计原则：

  • 初始容量应该 >= 预期最大元素数量 × 1.5（预留50%空间）
  • 当前业务量5万，设置为10万，预留2倍空间

6.2 效果验证

修复后的效果：

  1. 误判率降低：
    • 修改前：误判率可能达到10%+
    • 修改后：误判率保持在1%左右
  2. 循环次数减少：
    • 修改前：平均循环次数可能达到10+次
    • 修改后：平均循环次数接近1次（几乎不循环）
  3. Redis查询频率降低：
    • 修改前：3亿次/秒
    • 修改后：正常水平（大幅降低）
  4. 数据库连接池恢复正常：
    • 修改前：连接池被占满
    • 修改后：连接池使用正常

九、附录：布隆过滤器基础知识

9.1 布隆过滤器适用场景

适合使用：

  • 判断元素是否可能存在（允许误判）
  • 数据量大，需要节省内存
  • 查询频繁，需要快速响应

不适合使用：

  • 需要精确判断（不允许误判）
  • 需要删除元素（布隆过滤器不支持删除）
  • 数据量小（HashSet更合适）

9.2 布隆过滤器参数选择

参数关系：

误判率 ≈ (1 - e^(-kn/m))^k

其中：
- m：位数组长度（容量）
- k：哈希函数个数
- n：已插入元素个数

参数选择建议：

  • 误判率：通常选择0.01（1%）或0.1（10%）
  • 哈希函数个数k：通常选择3-5个
  • 容量m：根据预期元素数量n和误判率计算

在线计算工具：

  • https://hur.st/bloomfilter/
  • 输入预期元素数量和误判率，自动计算所需容量

9.3 布隆过滤器实现

Guava实现：

import com.google.common.hash.BloomFilter;
import com.google.common.hash.Funnels;

// 创建布隆过滤器
BloomFilter<String> bloomFilter = BloomFilter.create(
    Funnels.stringFunnel(Charset.defaultCharset()),
    100000,  // 预期插入元素数量
    0.01     // 误判率
);

// 添加元素
bloomFilter.put("element");

// 判断元素是否存在
boolean mightContain = bloomFilter.mightContain("element");
