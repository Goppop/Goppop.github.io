支付系统实现思路演进

实现一个支持多种支付渠道（微信、支付宝等）的支付系统

阶段一：最简单的实现（面向过程）

思路

直接写一个方法，根据渠道类型调用不同的支付逻辑

问题

  1. 代码膨胀：每增加一个支付渠道，都要加一个 if-else
  2. 难以维护：所有支付逻辑混在一起
  3. 违反开闭原则：修改代码才能添加新功能

阶段二：面向对象初步改进

思路

每个支付渠道单独一个类，用接口统一

改进

  • 每个支付渠道独立成类，职责清晰
  • 通过接口统一调用方式

问题

  1. 还是有 if-else：创建客户端时还是需要判断
  2. 没有统一流程：每个实现类都要自己写验证、异常处理
  3. 重复代码：每个实现类都有类似的逻辑

阶段三：引入工厂模式（策略模式雏形）

思路

用 Map 存储策略映射，消除 if-else

改进

  • 消除 if-else：通过 Map 查找，O(1) 时间复杂度
  • 易于扩展：新增渠道只需要在 Map 中添加一行
  • 符合开闭原则：对扩展开放，对修改关闭

问题

  1. 重复代码问题没解决：每个实现类都要写验证、异常处理
  2. 没有统一流程：不同实现类的流程可能不一致

阶段四：引入模板方法模式

思路

抽象类定义统一流程，子类实现具体逻辑

改进

  • 统一流程：所有支付渠道都遵循相同的执行步骤
  • 减少重复：验证、异常处理、日志等公共逻辑只在父类写一次
  • 职责清晰：子类只关注自己的支付逻辑

问题

  1. 工厂模式不完善：每次都创建新实例，没有缓存
  2. 配置管理：没有统一管理配置的更新

阶段五：完善工厂模式（加入缓存和配置管理）

思路

工厂管理客户端生命周期：创建、缓存、更新

改进

  • 性能优化：客户端实例缓存，避免重复创建
  • 配置管理：支持配置更新时刷新客户端
  • 线程安全：使用 ConcurrentHashMap

设计思路总结

我的实现思路（从零开始）

  1. 第一步：定义接口
    • 思考：支付客户端需要哪些方法？
    • 结果：PayClient 接口
  2. 第二步：实现具体类
    • 思考：每个支付渠道怎么实现？
    • 结果：WxNativePayClient、AlipayWapPayClient 等
  3. 第三步：发现重复代码
    • 问题：每个实现类都有验证、异常处理等重复代码
    • 解决：提取抽象类 AbstractPayClient，使用模板方法模式
  4. 第四步：消除 if-else
    • 问题：创建客户端时有很多 if-else
    • 解决：用 Map 存储策略映射，引入工厂模式
  5. 第五步：优化性能
    • 问题：每次都创建新实例，性能差
    • 解决：工厂管理缓存，复用实例
  6. 第六步：支持配置更新
    • 问题：配置更新时需要刷新客户端
    • 解决：工厂支持检测配置变化并刷新

设计模式的应用时机

  • 模板方法模式：当发现多个类有相同的执行流程时
  • 策略模式：当发现有很多 if-else 需要选择不同实现时
  • 工厂模式：当需要统一管理对象创建和生命周期时

关键设计原则

  1. 单一职责：每个类只做一件事
  2. 开闭原则：对扩展开放，对修改关闭
  3. 依赖倒置：依赖抽象，不依赖具体实现
