一、项目背景

1.1 业务场景

这是一个算力租赁平台的扣费系统，主要业务场景包括：

  1. 包月预付费：用户购买包月算力，一次性扣除整月费用
  2. 按小时后付费：用户按小时使用算力，每小时自动续扣
  3. 退款处理：支持全额退款、部分退款、按订单退款、按扣费单退款
  4. 余额预警：提前通知用户余额不足，避免服务突然中断

1.2 核心挑战

  • 资金安全：不能重复扣费，不能多退少退
  • 高并发：支持大量用户同时扣费
  • 幂等性：网络重试、系统故障恢复时不能重复扣费
  • 系统解耦：与订单系统、资源系统解耦，通过消息队列异步通信
  • 可追溯性：每笔扣费都有完整的流水记录，支持对账

4.2 核心设计理念

4.2.1 主记录 + 明细记录设计

设计思路：

  • 主记录（DeductionRecord）：记录订单级别的扣费配置，每个订单只有1条主记录
  • 明细记录（DeductionDetail）：记录每次实际扣费的详细信息，每次扣费产生1条明细

优势：

  • 支持细粒度退款（可以退某一次扣费）
  • 支持部分退款（可以退部分金额）
  • 便于对账和审计

关系说明：

  • 一次性扣费：1个主记录 + 1个明细记录
  • 周期扣费：1个主记录 + N个明细记录（N=实际扣费次数）

4.2.2 同步调用 vs 消息队列

最终选择：同步调用

原因：

  1. 实时性要求高：扣费结果需要立即返回，不能异步处理
  2. 事务一致性：扣费、余额扣减、流水记录需要在同一事务中完成
  3. 简单可靠：同步调用逻辑清晰，易于调试和排查问题
  4. 消息队列用于通知：扣费成功后通过消息队列通知订单系统，而不是用于扣费流程本身

消息队列的使用场景：

  • 扣费结果通知（扣费成功/失败）
  • 余额不足通知
  • 余额预警通知
  • 退款结果通知

5.1 扣费记录表（主记录）

关键设计点：

  1. 幂等键（idempotent_key）：

    • 格式：{orderNo}_{deductionType}_{period}
    • 一次性扣费：ORD001_MONTHLY_PREPAID_0
    • 周期扣费：ORD002_HOURLY_POSTPAID_PERIODIC（不带具体期数）

  2. 周期扣费字段：

    • is_periodic：标识是否为周期扣费
    • next_deduction_time：下次扣费时间，用于定时任务扫描
    • current_period：当前已扣费次数

  3. 状态字段：

    • status：完整的状态流转，支持失败重试和退款

5.2 扣费明细表

关键设计点：

  2. 退款字段：refunded_amount 和 refund_status 支持部分退款
  3. 冗余字段：order_no、instance_id 等冗余字段便于查询，避免关联查询

八、踩坑与优化

8.1 设计问题与改进

8.1.1 问题1：控制粒度太粗

问题描述：

  • 最初只支持按订单号退款和停止扣费
  • 升降配场景下，无法精确控制某个扣费单

解决方案：

  • 新增按扣费单号退款：refundByDeductionNo(deductionNo, reason)
  • 新增部分退款：partialRefund(deductionNo, refundAmount, reason)
  • 新增按扣费单号停止：stopPeriodicDeductionByDeductionNo(deductionNo)

8.1.2 问题2：周期扣费续扣时创建新主记录

问题描述：

  • 最初设计每次续扣都创建新的主记录
  • 导致一个订单有多条主记录，难以管理

解决方案：

  • 周期扣费只创建一条主记录
  • 后续续扣只创建明细记录，不创建新主记录
  • 主记录通过 current_period 和 next_deduction_time 跟踪续扣状态

8.1.3 问题3：退款并发冲突

问题描述：

  • 多个线程同时退款时，可能重复退款

解决方案：

  • 使用CAS更新退款金额
  • 更新失败时回滚余额并抛出异常

十、总结与思考

10.1 核心亮点

  1. 主记录+明细记录设计：

    • 支持细粒度退款和部分退款
    • 便于对账和审计

  2. 完善的幂等性保证：

    • 数据库唯一索引 + 应用层校验
    • 支持并发场景下的幂等性

  3. CAS更新防止并发冲突：

    • 周期续扣、退款等关键操作都使用CAS更新
    • 保证数据一致性

  4. 停机补扣机制：

    • 系统停机期间漏扣的周期自动补扣
    • 限制最多补扣10个周期，避免一次性扣太多

  5. 余额预警机制：

    • 提前3天和1天两级预警
    • Redis去重，避免重复发送

10.2 设计思考

10.2.1 为什么选择同步调用而不是消息队列？

原因：

  1. 实时性要求：扣费结果需要立即返回，不能异步处理
  2. 事务一致性：扣费、余额扣减、流水记录需要在同一事务中完成
  3. 简单可靠：同步调用逻辑清晰，易于调试和排查问题

消息队列的使用场景：

  • 扣费结果通知（扣费成功后通知订单系统）
  • 余额不足通知
  • 余额预警通知

10.2.2 为什么周期扣费只创建一条主记录？

原因：

  1. 便于管理：一个订单只有一条主记录，便于查询和管理
  2. 状态跟踪：通过 current_period 和 next_deduction_time 跟踪续扣状态
  3. 退款方便：退款时只需要操作一条主记录

明细记录的作用：

  • 记录每次实际扣费的详细信息
  • 支持细粒度退款（可以退某一次扣费）

10.3 未来优化方向

  1. 性能优化：

    • 批量处理续扣任务
    • Redis缓存热点数据
    • 数据库分表（按月分表）

  2. 功能增强：

    • 支持更多计费模式（按天、按周）
    • 支持优惠券抵扣
    • 支持欠费处理

  3. 监控完善：

    • 对接监控系统（Prometheus + Grafana）
    • 配置告警规则
    • 定期生成对账报表

10.4 经验总结

  1. 设计阶段要充分考虑业务场景：

    • 升降配、部分退款等复杂场景要在设计阶段考虑清楚

  2. 幂等性是金融系统的生命线：

    • 所有涉及资金的操作都必须保证幂等性
    • 数据库唯一索引 + 应用层校验是双重保障

  3. 并发控制要使用CAS而不是锁：

    • CAS更新性能更好，不会阻塞其他线程
    • 更新失败时要有回滚机制

  4. 异常处理要完善：

    • 业务异常和系统异常要分别处理
    • 失败时要记录详细日志，便于排查问题

  5. 测试要充分：

    • 单元测试、集成测试、压力测试都要覆盖
    • 特别是并发场景和异常场景的测试
